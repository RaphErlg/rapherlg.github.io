Lab_sys_03 Heartbeat


But et topo

Le but de la manip est de montrer qu’un site web reste accessible même si une des deux machines tombe, sans changer l’adresse IP utilisée dans le navigateur.

La topologie demandée est composée de deux VM Debian sur le même Réseau NAT VirtualBox, avec une IP virtuelle de cluster prévue en 10.40.20.100/24.

Dans ton cas, les deux VM ont reçu des IP dynamiques finissant en .3 et .4, ce qui colle avec le fonctionnement attendu du réseau NAT du cours.

Il faut garder en tête que .3 et .4 sont les IP normales des VM, tandis que 10.40.20.100 est l’IP virtuelle portée par le cluster.

Préparation

Le support demande d’abord de nommer les deux machines debian1 et debian2, puis de compléter /etc/hosts pour que chaque machine connaisse l’autre sans DNS.

Il demande ensuite de vérifier la résolution de noms avec ping, puis d’installer apache2, apache2-utils et heartbeat.

Sur debian1, tu fais ceci :

nano /etc/hostname
debian1
nano /etc/hosts
127.0.0.1       localhost
127.0.1.1       debian1
10.40.20.4      debian2

Sur debian2, tu fais ceci :

nano /etc/hostname
debian2
nano /etc/hosts
127.0.0.1       localhost
127.0.1.1       debian2
10.40.20.3      debian1

Ensuite, sur les deux machines, tu installes les paquets demandés.

apt-get update
apt-get install apache2 apache2-utils heartbeat

Apache

La partie Apache sert à préparer le service web sur les deux nœuds avant la haute disponibilité, avec un VirtualHost commun et une page différente sur chaque machine pour voir immédiatement quel serveur répond.

Le document demande de créer webapp.conf, de désactiver le site par défaut, d’activer ce nouveau site, puis de créer /var/www/index.html avec Site1 sur debian1 et Site2 sur debian2.

Sur les deux machines, crée ce fichier :

nano /etc/apache2/sites-available/webapp.conf
text<VirtualHost *:80>
    ServerName debian-cluster
    ServerAdmin webmaster@localhost

    DocumentRoot /var/www
    <Directory />
        Options FollowSymLinks
        AllowOverride all
    </Directory>
    <Directory /var/www/>
        Options Indexes FollowSymLinks MultiViews
        AllowOverride all
        Order allow,deny
        allow from all
    </Directory>

    ScriptAlias /cgi-bin/ /usr/lib/cgi-bin/
    <Directory "/usr/lib/cgi-bin">
        AllowOverride None
        Options +ExecCGI -MultiViews +SymLinksIfOwnerMatch
        Order allow,deny
        Allow from all
    </Directory>

    ErrorLog ${APACHE_LOG_DIR}/error.log
    LogLevel warn
    CustomLog ${APACHE_LOG_DIR}/access.log combined
</VirtualHost>

Puis active cette configuration sur les deux machines.

a2dissite 000-default
a2ensite webapp.conf
service apache2 restart

Ensuite, crée les pages de test, car DocumentRoot /var/www dit à Apache d’aller chercher le site dans ce dossier.

Sur debian1 :

echo '<html><p>Site1</p></html>' | sudo tee /var/www/index.html

Sur debian2 :

echo '<html><p>Site2</p></html>' | sudo tee /var/www/index.html

Enfin, vérifie qu’Apache a bien compris ta configuration avec cette commande, qui affiche les VirtualHosts actifs.

bashapache2ctl -S

Heartbeat

La partie Heartbeat sert à définir la ressource du cluster, la manière dont les nœuds se surveillent, et le mécanisme d’authentification entre eux.

Le support demande trois fichiers, haresources, ha.cf et authkeys, puis le démarrage du service et la lecture du log /var/log/ha-log.

Sur les deux machines, crée haresources avec la ligne utilisée dans ta manip, qui associe le nœud préféré, l’IP virtuelle, l’interface réseau et le service Apache.

nano /etc/ha.d/haresources
debian1 IPaddr::10.40.20.100/24/enp0s3 apache2

Sur les deux machines, crée ensuite ha.cf, qui contient les règles de dialogue entre les nœuds.

nano /etc/ha.d/ha.cf
logfile /var/log/ha-log
logfacility local0
keepalive 2
deadtime 30
initdead 120
bcast enp0s3
udpport 694
auto_failback on

node debian1
node debian2

Sur les deux machines, crée enfin authkeys, puis applique les permissions demandées.

nano /etc/ha.d/authkeys
auth 2
2 sha1 test-ha
chmod 600 /etc/ha.d/authkeys

Le sens de ces fichiers est simple : haresources définit quoi basculer, ha.cf définit comment les nœuds se surveillent, et authkeys permet aux deux machines de se faire confiance.

Le paramètre important pour les tests est deadtime 30, qui explique pourquoi le basculement n’est pas immédiat après la panne de debian1.

Pour démarrer et vérifier Heartbeat, fais sur les deux machines :

service heartbeat start
cat /var/log/ha-log

Tests

Le test final du support consiste à ouvrir http://10.40.20.100, vérifier que Site1 apparaît d’abord, couper debian1, constater une brève indisponibilité, puis attendre environ 30 secondes pour voir Site2.

Le document insiste aussi sur le fait qu’il faut vider le cache du navigateur avant chaque essai pour ne pas fausser l’observation.

L’ordre de test à refaire est celui-ci :

  1. Vérifier que http://10.40.20.100 affiche Site1.

  2. Éteindre debian1 pour simuler une panne.

  3. Recharger la page, constater qu’elle peut devenir temporairement indisponible.

  4. Attendre environ 30 secondes.

  5. Recharger à nouveau et vérifier que Site2 apparaît.

Ce résultat prouve que ton cluster web Heartbeat remplit bien l’objectif du labo : le service web reste disponible via la même IP virtuelle malgré la perte du premier nœud.
