Architecture des Systèmes Q2


Synthèse couvrant la présentation du cours (chap. 01) et le chapitre de rappels (chap. 02).

1. Présentation du cours

1.1 Objectifs (acquis d'apprentissage)

Au terme de l'activité d'apprentissage, l'étudiant sera capable de :

  • décrire le fonctionnement de la mémoire, d'un processeur et de ses microarchitecture et ISA

  • expliquer les différents types de virtualisation

  • expliquer les concepts associés à la sauvegarde d'informations

  • identifier les défaillances d'un système d'informations et proposer des solutions de correction

1.2 Plan du cours

Le cours suit une approche en couches d'abstraction, du matériel vers le logiciel :

  1. Introduction et rappels

  2. Couche Logique (mémoires, bus, microprocesseurs)

  3. Couche microarchitecture

  5. Architectures parallèles

3. Modèles d'architecture

3.1 Modèle général

Un ordinateur est composé de :

  • une Unité Centrale (CPU) contenant :

    • une ALU (Unité Arithmétique et Logique) pour les calculs

    • une Unité de Commande (Control Unit) pour orchestrer les opérations

  • une Mémoire pour stocker programme et données

  • des Entrées/Sorties pour communiquer avec l'extérieur

│       Unité Centrale (CPU)          │
│  ┌─────────┐      ┌──────────────┐  │
│  │   ALU   │      │ Unité de     │  │
│  │         │      │ Commande     │  │

           │
       Mémoire ←→ Entrées / Sorties

3.2 Von Neumann

  • Unité de traitement (ALU) et unité de commande séparées dans le CPU

  • Mémoire commune pour les instructions ET les données

  • Modèle dominant des ordinateurs modernes

3.3 Harvard

  • Unité de traitement et unité de commande séparées

  • Mémoires séparées pour les instructions et pour les données

  • Permet des accès simultanés instruction/donnée → plus rapide

  • Utilisé dans certains DSP, microcontrôleurs, et au niveau des caches L1 modernes

Les deux familles coexistent aujourd'hui : CISC domine les PC/serveurs (x86), RISC domine le mobile et l'embarqué (ARM) et progresse sur les serveurs.

4. Composants d'un système

4.1 Le CPU (Central Processing Unit)

Deux grandes fonctions internes :

  • Séquenceur (Control Unit) : reçoit les instructions, génère les signaux de commande

  • Unité de calcul (ALU) : traite les données

Le CPU communique avec l'extérieur via le bus (instructions et données).

4.2 Les E/S (Entrées / Sorties)

Périphériques de communication avec l'utilisateur ou d'autres systèmes (clavier, écran, disque, réseau…). Reliés au CPU via des unités d'échange (I/O controllers) qui pilotent les périphériques.

4.4 Les bus

Un bus est un ensemble de fils transportant des signaux entre composants. Trois grandes catégories de bus principaux :

  • Bus d'adresses : indique au composant cible où lire/écrire

  • Bus de données : transporte les données

  • Bus de contrôle : signaux de commande (lecture, écriture, horloge…)

5. Exécution d'un programme

5.1 Les étapes (du code source à l'exécution)

  1. Programme source écrit en langage haut niveau (ex. C)

  2. Compilation → langage assembleur (ex. MIPS : multi, add, lw, sw, jr…)

  3. Assemblage → code machine binaire

  4. Stockage de l'exécutable sur le disque dur

  5. Chargement du programme du disque vers la RAM au lancement

  6. Exécution par le CPU, instruction par instruction, en passant par les caches L1 → L2 → L3

5.2 Exécution d'une instruction par le CPU

Chaque instruction suit un cycle en 4 étapes :

Étape

Acteur

Rôle

1. Fetch

Control Unit

Va chercher l'instruction en mémoire (à l'adresse pointée par le PC)

2. Decode

Control Unit

Décode l'instruction en commandes élémentaires pour l'ALU

3. Execute

Ce cycle Fetch → Decode → Execute → Store est la base de tout le fonctionnement du processeur. Il sera détaillé dans la note Microarchitecture.

6. À retenir

  • Un système informatique se conçoit par couches d'abstraction (de l'électron au programme).

  • Le CPU contient une ALU (calculs) et une Control Unit (orchestration).

  • Deux grands modèles d'architecture : Von Neumann (mémoire unique) et Harvard (mémoires séparées).

  • Deux grands styles d'ISA : CISC (Intel x86) et RISC (ARM, MIPS).

  • Composants reliés par 3 bus principaux : adresses, données, contrôle.

  • L'exécution d'une instruction suit le cycle Fetch → Decode → Execute → Store. -e

Synthèse couvrant les chapitres Mémoires (03), Bus (04) et Microprocesseurs (05). La couche logique regroupe les composants matériels concrets qui mettent en œuvre le calcul : mémoires, bus, processeur, périphériques.

PARTIE A — Les Mémoires

A.1 Généralités

Rôle des mémoires dans un système

  • Le programme et les données sont enregistrés en mémoire centrale (RAM) pour être exécutés par le CPU.

  • Le processeur utilise des registres, qui sont aussi des zones de stockage internes.

  • Au démarrage, un programme préenregistré dans une mémoire ROM charge le noyau du système d'exploitation en RAM.

  • Pour harmoniser les vitesses entre processeur et RAM, on intercale une mémoire cache plus rapide.

  • Les mémoires de masse (HDD, SSD…) conservent durablement de grandes quantités d'informations.

A.2 Caractéristiques

Quatre caractéristiques principales définissent une mémoire :

1. La capacité

Quantité d'information qu'une mémoire peut stocker (en bits, octets, Ko, Mo, Go…).

2. Le temps d'accès

Temps nécessaire pour obtenir une information.

  • Mémoires électroniques (RAM, ROM, registres) : nanosecondes (10⁻⁹ s)

  • Mémoires magnétiques ou optiques (HDD, CD) : millisecondes (10⁻³ s)

Le temps d'accès peut aussi s'exprimer comme une fréquence :

f=1taccf = \frac{1}{t_{acc}}f=tacc​1​

Exemple : un temps d'accès de 10 ns ↔ une fréquence de 100 MHz.

3. La bande passante

Quantité d'informations transférée par unité de temps (Mo/s, Go/s…).

4. Le temps de latence

Nombre de cycles d'horloge perdus pour réaliser une opération.

Exemple : une latence de 3 signifie que l'obtention de l'information coûte 3 cycles d'horloge (cycles « perdus »).

A.3 Les types de mémoires

On classe les mémoires en deux grandes catégories selon leur rôle :

  • Mémoires de travail : actives pendant l'exécution d'un programme (RAM, cache, registres)

  • Mémoires de stockage : conservent durablement de grandes quantités (HDD, SSD…)

A.3.1 Les mémoires vives (RAM)

Volatiles : perdent leur contenu sans alimentation électrique.

DRAM (Dynamic RAM)

  • Cellule de base : 1 condensateur + 1 transistor par bit

  • L'information est stockée comme une charge électrique dans le condensateur

  • Doit être rafraîchie toutes les quelques millisecondes (sinon perte de la charge)

  • C'est la base des mémoires centrales des ordinateurs

  • SDRAM : variante synchronisée avec l'horloge du système

  • DDR SDRAM (Double Data Rate) : produit des données sur les fronts montant ET descendant de l'horloge → débit doublé

  • Versions successives : DDR, DDR2, DDR3, DDR4…

SRAM (Static RAM)

  • Cellule de base : une bascule de transistors (4 à 6 transistors par bit)

  • Pas de rafraîchissement nécessaire

  • Très rapide (6–15 ns) mais chère et peu dense

  • Utilisée pour les mémoires cache (faible capacité)

A.3.2 Les mémoires mortes (ROM family)

Non volatiles : conservent leur contenu sans alimentation.

Type

Caractéristiques

Contenu figé à la fabrication. Stockage permanent. Coût élevé, réservé aux grandes séries.

Écriture unique par l'utilisateur, via une machine spéciale. Utilise des fusibles détruits lors de la programmation.

A.4 Fonctionnement

A.4.1 Organisation matricielle

Une mémoire est un tableau de cellules :

  • Lignes → mots (words)

  • Colonnes → bits dans chaque mot

On sélectionne une ligne entière via une adresse déposée sur le bus.

A.4.2 Comment un bit est-il stocké ?

Type

Mécanisme de stockage

SRAM

Bit stocké dans un circuit bistable ; la valeur reste tant que l'alimentation est présente. Pas de rafraîchissement.

DRAM

Charge électrique dans un condensateur ; doit être rafraîchie périodiquement.

ROM

Aucun état électrique stocké : le bit est défini physiquement (transistor présent → 0, transistor absent → 1). Contenu figé à la fabrication.

A.4.3 Dialogue CPU ↔ Mémoire centrale

La RAM dialogue avec le processeur via les bus (adresses, données, contrôle).

Lecture par le processeur :

  1. Dépose l'adresse sur le bus d'adresses

  2. Active la commande de lecture

  3. Lit la donnée sur le bus de données

Écriture par le processeur :

  1. Dépose l'adresse sur le bus d'adresses

  2. Dépose la donnée sur le bus de données

  3. Active la commande d'écriture

A.4.4 Les mémoires caches

Problème : la fréquence du processeur (GHz) est très supérieure à celle de la DRAM, et le temps de transfert par le bus devient un facteur de ralentissement majeur.

Principe du cache : faire coopérer des mémoires petites et très rapides, proches du processeur, avec des mémoires plus grandes et plus lentes, plus éloignées.

Fonctionnement :

  • Les mots les plus fréquemment utilisés sont placés dans le cache

  • Le processeur cherche d'abord dans le cache → si présent, lecture rapide

  • Si absent, accès à la mémoire centrale (plus lente) + copie du mot dans le cache

  • L'efficacité dépend du taux de succès (cache hit rate)

  • La cohérence cache ↔ mémoire centrale doit être maintenue : toute écriture dans le cache doit être répercutée en RAM

A.4.5 Les mémoires de masse

Disque magnétique (HDD)

  • Conserve les données de façon permanente, même éteint

  • Stocke : OS, logiciels, fichiers

  • Fonctionnement : plateaux magnétiques tournant à grande vitesse

  • Données enregistrées comme des zones magnétisées

  • Une tête de lecture/écriture (bobinage fer/nickel) flotte au-dessus du plateau :

    • Écriture : un courant dans la bobine magnétise la surface dans un sens (0 ou 1)

    • Lecture : le passage sur une zone magnétisée induit un champ électrique dans la bobine, dont le sens donne la valeur du bit

A.5.2 Organisation interne (matrice)

Pour une capacité donnée, plusieurs brochages sont possibles. Exemple : une mémoire de 64 kbits peut être organisée comme une matrice 256 × 256 bits ; les broches A0–A7 adressent soit une ligne (L) soit une colonne (C), la broche D fournit la donnée.

Opérations :

  • Lecture : adresse ligne → activer L → adresse colonne → activer C → activer lecture → lire le bit

  • Cette organisation minimise le nombre de broches mais ralentit les opérations

  • Les mémoires de grande capacité sont souvent organisées en matrice pour diminuer les coûts

A.5.3 Le décodeur d'adresses

Problème : le CPU a beaucoup plus de bits d'adresses qu'un seul boîtier mémoire.

Solution : un circuit spécialisé, le décodeur d'adresses (ou sélecteur), utilise les bits supplémentaires pour sélectionner le bon boîtier mémoire.

Exemple :

  • Processeur avec 17 bits d'adresses → adresse 2¹⁷ = 131 072 bytes

  • Chaque boîtier mémoire fait 2¹³ = 8 192 bytes

  • Le processeur peut adresser 2⁴ = 16 boîtiers

Une adresse mémoire correspond donc à :

  • un numéro de boîtier (4 bits supérieurs → décodeur)

  • un numéro de mot dans le boîtier (13 bits inférieurs)

Le signal « Memory ready » synchronise la mémoire et le processeur.

PARTIE B — Les Bus

B.1 Généralités

Définition

Un bus est un dispositif de transmission de données partagé entre plusieurs composants d'un système numérique.

Principe maître/esclave

Le maître initie la communication et donne les ordres ; l'esclave répond.

  • En général, le CPU est le maître et la mémoire/E/S sont les esclaves

  • Mais des composants d'E/S peuvent aussi devenir maîtres (ex. pour générer des interruptions)

B.2 Caractéristiques d'un bus

Quatre aspects principaux, qui ont tous un impact sur la vitesse et la bande passante :

  1. La largeur du bus

  2. La synchronisation (horloge)

  3. L'arbitrage

  4. Les opérations

B.2.1 La largeur du bus

Nombre de lignes parallèles → nombre de bits transportés simultanément.

Un bus 64 bits transporte 8 octets par cycle, soit deux fois plus qu'un bus 32 bits.

B.2.2 La synchronisation

Deux catégories de bus :

Bus synchrone

  • Possède une ligne d'horloge pilotée par un oscillateur à quartz (5 à 133 MHz typiquement)

  • Toutes les activités s'effectuent sur un nombre entier de cycles (cycles de bus)

  • La mémoire peut être plus lente → on insère des états d'attente (wait states)

  • Inconvénients :

    • Un transfert qui prendrait 3,1 cycles doit être étendu à 4 cycles (cycles fractionnaires interdits)

    • Avec des dispositifs hétérogènes, le bus doit être réglé sur le plus lent

Bus asynchrone

  • Pas d'horloge gérant les activités

  • Communication par poignée de main (handshake) entre maître et esclave

  • Chaque dispositif fonctionne à sa propre vitesse

B.2.3 L'arbitrage du bus

Problème : que se passe-t-il si plusieurs dispositifs veulent devenir maîtres en même temps ? Il faut un mécanisme d'arbitrage pour éviter les collisions.

Arbitrage centralisé

  • Un arbitre unique décide qui devient maître

  • Daisy chaining : la permission passe d'un dispositif à l'autre en série ; le premier demandeur le plus proche de l'arbitre l'emporte

  • Beaucoup de CPU ont l'arbitre intégré dans la puce

  • Variante à priorités multiples : pour chaque niveau de priorité, une ligne bus request et une ligne bus grant

Arbitrage décentralisé

  • Pas d'arbitre central : les dispositifs décident eux-mêmes en observant le bus

  • Chaque dispositif a sa propre ligne de demande

  • Inconvénient : beaucoup de fils, nombre limité de dispositifs

Arbitrage décentralisé à 3 lignes (exemple) :

  • Bus Request : « je veux le bus »

  • BUSY : « le bus est occupé »

  • Arbitrage (IN/OUT) : pour décider qui devient maître

Les dispositifs sont reliés en chaîne ; le plus proche du début qui demande gagne :

  • Si IN est désactivé → ne peut pas devenir maître

  • Si IN est activé et qu'il veut le bus → désactive OUT

  • Les autres se bloquent automatiquement et attendent

B.2.4 Les opérations du bus

Cycles ordinaires

Le maître (CPU) lit ou écrit des données depuis un esclave (mémoire).

Transferts par blocs

Utiles avec les caches : au lieu de transférer un mot, on transfère un bloc entier.

  • Le maître indique le nombre de mots à transférer

  • L'esclave fournit un mot à chaque cycle jusqu'à la fin

  • Un signal BLOCK indique qu'un transfert par bloc est demandé

Gestion des interruptions

Quand un périphérique d'E/S termine une tâche, il active une interruption → utilise le bus pour signaler la fin et déclencher le traitement par le CPU.

Exemple : le contrôleur 8259A peut gérer jusqu'à 8 dispositifs d'E/S, connectés à 8 entrées IRx (IR0 = Clock, IR1 = Keyboard, IR4 = Disk, IR7 = Printer…).

B.3.3 Le bus PCI Express (PCIe)

Problèmes du PCI

  • De plus en plus de périphériques trop rapides pour PCI

  • Augmenter la fréquence aggrave les problèmes (skew, crosstalk, effets capacitifs)

  • Cartes PCI grandes et rigides → inadaptées aux portables et mobiles

Caractéristiques du PCIe

  • Abandon du bus parallèle au profit de connexions série point à point à haute vitesse

  • Ce n'est plus vraiment un bus mais un réseau interne :

    • Un switch central (comme un switch Ethernet)

    • Chaque périphérique a une connexion directe et dédiée vers ce switch

    • Données circulent en paquets, pas en signaux partagés

  • PCIe : jusqu'à 64 Go/s ↔ PCI à 133 Mo/s (dizaines à centaines de fois plus rapide)

  • Vitesse dépend de la génération (1.0, 2.0…) et du nombre de lignes (×1, ×4, ×8, ×16)

Analogie : PCI = une seule route où tout le monde roule ensemble. PCI Express = un réseau d'autoroutes privées avec des échangeurs (switches).

B.3.5 Le bus USB

Objectif

Connecter facilement des périphériques d'E/S à faible vitesse → norme officielle en 1998.

  • Un seul type de câble pour presque tout

  • Organisation en arbre avec hubs (concentrateurs)

  • Communication contrôlée par l'ordinateur (pas entre périphériques)

Câble

4 fils : 2 données + 1 alimentation (+5 V) + 1 masse.

Topologie

Évolution

PARTIE C — Les Microprocesseurs

C.1 Généralités

Un CPU sur une puce

Tous les CPU modernes sont intégrés sur une seule puce (chip). Chaque puce CPU possède un ensemble de broches (pins) par lesquelles transitent toutes les communications avec l'extérieur. Comprendre la fonction des broches = comprendre comment le CPU interagit avec mémoire et E/S.

Les types de broches

Les broches d'une puce CPU se classent en 3 types principaux + 2 catégories spéciales :

1) Broches d'adresses (Address pins)

  • Indiquent où lire ou écrire en mémoire

  • Avec m broches d'adresses, le CPU peut accéder à 2ᵐ adresses mémoire

Nombre de broches

Adresses accessibles

16 bits

65 536 (~64 Ko)

32 bits

4 milliards (4 Go)

64 bits

Espace immense

Plus il y a de broches d'adresses, plus on peut utiliser de mémoire RAM.

2) Broches de données (Data pins)

Transportent les données entre le CPU et la mémoire/E/S. Leur nombre détermine la taille des mots transférés.

3) Broches de contrôle (Control pins)

Indiquent ce que le CPU veut faire :

  • Lire la mémoire

  • Écrire en mémoire

  • Accéder à un périphérique

  • Démarrer une opération

Le CPU dirige tout le système grâce à ces signaux.

Broches d'interruption

  • Entrées des dispositifs d'E/S vers le CPU

  • Le CPU peut ordonner à une E/S de démarrer une opération puis faire autre chose

  • Quand l'opération est terminée, le contrôleur d'E/S active un signal d'interruption

  • Certains CPU ont une broche de sortie pour acquitter l'interruption

Broches d'arbitrage du bus

  • Régulent le trafic sur le bus pour empêcher deux dispositifs de l'utiliser en même temps

  • Le CPU est considéré comme un dispositif à part entière et doit demander l'accès au bus

Broches auxiliaires

  • Coprocesseurs (FPU, GPU…)

  • Information d'état, débogage, réinitialisation

  • Compatibilité avec d'anciens périphériques

Liaison aux E/S

Les opérations d'E/S échangent des informations entre composants internes et périphériques via des unités d'échange (I/O controllers). Deux manières d'atteindre un circuit d'E/S :

  1. Adressage spécifique : un signal sur le bus indique qu'on fait une opération d'E/S

  2. Adressage commun à la mémoire : les E/S font partie de l'espace d'adressage du CPU ; on adresse une E/S comme on adresse la mémoire (un circuit de sélection prend en charge cette portion d'adresse)

C.2 L'architecture Intel Core i7

C.2.1 Évolution historique

  • Descendant direct du 8088 (PC IBM original)

  • Premier Core i7 lancé en novembre 2008 : quadricœur, 731 millions de transistors, jusqu'à 3,2 GHz, gravure 45 nm

  • Largeur de gravure = largeur des fils reliant les transistors. Plus elle est faible, plus on peut intégrer de transistors sur une puce.

C.2.2 Multicœurs et hyperthreading

Multicœurs

  • Le Core i7 contient plusieurs processeurs sur la même puce (2 à 6)

  • Approche pour augmenter les performances sans augmenter la fréquence

  • Un cœur physique = circuits complets pour exécuter des instructions (registres, CU, ALU)

  • Architectures :

    • Symétriques : tous les cœurs identiques

    • Asymétriques : certains cœurs dédiés (affichage, audio…)

Hyperthreading

  • Chaque cœur peut exécuter plusieurs threads en même temps

  • Permet de masquer les temps morts (ex. accès mémoire)

  • Plus efficace que le multitâche purement logiciel

  • Les programmeurs doivent écrire des programmes parallèles (threads + locks) pour profiter pleinement

  • Cache partagé ou non entre cœurs : un cache partagé peut diminuer les performances (latence + conflits d'accès)

C.2.3 Le pipeline

  • Permet d'exécuter plusieurs étapes d'instructions en parallèle

  • Profondeur du pipeline du Core i7 : 14 à 16 étages

  • Intel privilégie le parallélisme (superscalaire) plutôt qu'un pipeline très profond

Architecture des Systèmes 2 — Note 3 : Microarchitecture

Synthèse du chapitre 06 — Couche microarchitecture. On y apprend comment les briques matérielles de la couche logique (registres, ALU, mémoires, multiplexeurs) sont agencées pour exécuter concrètement les instructions d'une ISA. Le cours utilise l'architecture MIPS 32 bits comme cas d'étude.

1. Généralités

1.1 Définition

Une microarchitecture est une implémentation matérielle (circuits logiques) d'une ISA (jeu d'instructions) spécifique.

Vue d'ensemble interne d'un CPU (microarchitecture typique) :

  • Bus interne de données + bus interne d'adresses

  • Registres généraux (R0…Rn), registre de données, registre d'adresses, registre d'état, registre d'instruction (Code Op. + Adresse)

  • PC (Program Counter), SP (Stack Pointer)

  • ALU + décodeur

  • Bloc Logique de Commande (CU) piloté par l'horloge et par les signaux internes

  • Connexion vers les bus externes : bus de données, bus d'adresses, bus de commande, accumulateur

1.2 Rôle

La microarchitecture est l'agencement spécifique des registres, ALU, machines à états finis (FSM), mémoires et autres blocs logiques nécessaires pour implémenter une ISA.

Points clés :

  • Une même ISA peut posséder plusieurs microarchitectures différentes, chacune avec ses compromis (performances / coût / complexité)

  • Elles exécutent toutes les mêmes programmes, mais leurs conceptions internes peuvent varier considérablement

  • C'est le domaine du Logic Design (conception des systèmes numériques)

1.3 But

Une microarchitecture = un ensemble de fonctions logiques qui réalisent concrètement les opérations décrites par l'ISA.

1.4 Implémentation

Deux grands blocs

Bloc

Rôle

Chemin de données (datapath)

Exécute les opérations sur les données. Contient mémoires, registres, ALU, multiplexeurs. Opère sur des mots de largeur fixe (ex. 16, 32, 64 bits selon l'architecture).

Unité de contrôle (control unit)

Reçoit l'instruction courante depuis le datapath et lui indique comment l'exécuter. Génère les signaux de sélection des multiplexeurs, d'activation des registres et d'écriture mémoire. C'est le cerveau qui ordonne au datapath ce qu'il faut faire.

Le chemin de données — vue logique

Il organise la vue logique du processeur en fonction des traitements appliqués aux données :

  1. Instruction lue en mémoire

  2. Instruction décodée = code + data

  3. Calcul effectué sur les données

  4. Accès donnée(s) en mémoire

  5. Stockage du résultat en mémoire

Simplifications retenues dans le cours

Pour rester abordable, le cours fait 6 hypothèses :

  1. Architecture générale MIPS 32 bits

  2. Limitation à quelques instructions : load, store, add, sub, branch

  3. Les étapes pour exécuter une instruction sont identiques

  4. Un seul (long) cycle d'horloge par instruction (toutes les opérations en un cycle)

  5. À chaque cycle d'horloge, une instruction est toujours lue

  6. Pipeline RISC à 5 étages

Les 5 étapes d'exécution d'une instruction

1. FETCH   → 2. DECODE → 3. EXECUTE → 4. MEM ACCESS → 5. WRITE BACK
   (IF)        (ID)        (EX)          (MEM)            (WB)

Étape

Action

1. Fetch (IF)

Envoyer le PC vers la mémoire d'instructions, lire l'instruction, incrémenter PC (PC + 4 en MIPS 32 bits).

2. Decode (ID)

Extraire le code de l'opération, lire 1 ou 2 registres dans le register file selon l'instruction.

3. Execute (EX)

Effectuer l'opération via l'ALU (calcul, comparaison, calcul d'adresse pour load/store ou branch).

4. Memory access (MEM)

Si load : lire la donnée en mémoire. Si store : écrire en mémoire. Pour branch : éventuellement modifier le PC selon le résultat de la comparaison.

5. Write back (WB)

Écrire le résultat (ALU ou mémoire) dans un registre.

Pour une addition ADD R1, R2, R3, c'est à la fin de l'étape WB que la somme est écrite dans R1.

2. Conception du chemin de données

2.1 Blocs de base

Le chemin de données = unité utilisée pour traiter ou stocker des données à l'intérieur d'un processeur. On le conçoit en examinant les principaux composants nécessaires à chaque type d'instruction.

2.2 FETCH — Aller chercher l'instruction

Trois éléments combinés :

  • Le PC (Program Counter) : registre qui contient l'adresse de l'instruction courante

  • La mémoire d'instructions (Instruction memory) : on l'adresse avec le PC pour lire l'instruction

  • Un additionneur (Add) : pour calculer PC + 4 (architecture 32 bits → instructions de 4 octets) et préparer l'adresse suivante

        +----+
PC ───→ |Read|       Instruction
        |addr|─────→ memory ──────→ Instruction (32 bits)
        +----+         │
            │          │
            └──→ +4 ──→ Add ──→ (nouvelle valeur du PC)

Cet additionneur peut être construit à partir de l'ALU, simplement en câblant ses lignes de contrôle pour qu'elle effectue toujours une addition.

2.3 DECODE — Décoder et lire les registres

Le register file

Les registres généraux sont regroupés dans une structure appelée fichier de registres (register file) :

  • N'importe quel registre peut être lu ou écrit en spécifiant son numéro

  • Mécanismes de sélection pour lire/écrire plusieurs registres simultanément

Pour une instruction comme ADD, il faut :

  • Lire 2 mots depuis les registres → 2 entrées (numéros à lire), 2 sorties (données lues)

  • Écrire 1 mot dans un registre → 1 entrée (numéro à écrire), 1 entrée (donnée à écrire), 1 signal d'écriture (RegWrite) qui doit être activé au front d'horloge

Le register file a donc typiquement 4 entrées + 2 sorties + 1 signal de contrôle.

Instruction ADD — utilisation de l'ALU

Pour ADD, on lit 2 registres puis on passe leurs valeurs dans l'ALU, qui effectue l'opération et fournit le résultat (et un signal Zero indiquant si le résultat est nul).

Instruction LOAD — lw $t0, 8($t1)

Que fait l'instruction ?

  1. Prendre le contenu du registre $t1

  2. Ajouter l'offset signé 8

  3. Lire la mémoire à l'adresse [8 + $t1]

  4. Écrire la donnée lue dans le registre $t0

Instruction STORE — sw $t0, 8($t1)

Que fait l'instruction ?

  1. Lire le contenu du registre $t1

  2. Ajouter l'offset signé 8

  3. Lire le registre $t0

  4. Écrire la donnée lue dans $t0 à l'adresse [8 + $t1]

L'unité mémoire (Data Memory)

Avec load/store, on a besoin d'une unité mémoire distincte de la mémoire d'instructions, avec :

  • Une entrée Address

  • Une entrée Write data (pour les stores)

  • Une sortie Read data (pour les loads)

  • Deux signaux de contrôle : MemRead et MemWrite

   Address ───→ │                  │
                │   Data memory    │──→ Read data
Write data ───→ │                  │
                +──────────────────+
                  ↑              ↑
              MemWrite       MemRead

2.4 ALU — Branchement conditionnel (BEQ)

BEQ = Branch if EQual (MIPS) :

beq $t1, $t2, offset

L'instruction comporte 3 opérandes :

  • 2 registres ($t1, $t2) dont le contenu est comparé

  • Un décalage (offset) de 16 bits permettant de calculer l'adresse cible relative à l'instruction de branchement

Le datapath du branchement doit faire 2 opérations :

  1. Calculer l'adresse cible du branchement

  2. Comparer le contenu des registres

L'offset est signé, étendu en signe sur 32 bits, puis décalé de 2 bits (shift left 2) pour produire une adresse alignée sur un mot, puis ajouté à PC + 4 → c'est l'adresse de branchement.

Comment l'ALU fait la comparaison ?

Dans l'architecture MIPS étudiée, l'ALU fournit un signal Zero en sortie indiquant si le résultat est nul. La comparaison se fait donc en configurant une soustraction :

  • Si Zero est activé → les deux valeurs sont égales → on branche

3. Conception de la fonction de contrôle

3.1 Principe

Le datapath, à ce stade, sait :

  • Lire des registres

  • Utiliser l'ALU

  • Accéder à la mémoire

  • Calculer PC + 4

  • Calculer le PC pour les sauts conditionnels

La fonction de contrôle ajoute la capacité à :

  • Interpréter l'instruction lue en mémoire

  • Générer tous les signaux de commande nécessaires pour faire fonctionner correctement le datapath

Le datapath fait le travail. La fonction de contrôle lui dit quoi faire.

Ce qu'ajoute concrètement la fonction de contrôle

  • De la logique combinatoire

  • Qui oriente les données (via les multiplexeurs)

  • Et active/désactive les blocs matériels

💡 Le même matériel peut exécuter plusieurs instructions, uniquement en changeant les signaux de contrôle. La fonction de contrôle transforme une instruction en une configuration du datapath.

2.4 Exemple d'exercice — Traduire un programme

Énoncé : traduire le code Python suivant en assembleur de l'architecture du simulateur.

    n = n + 1
    x = x + x

À la fin du programme, x = 512 (c'est-à-dire 2⁹, puisque x est doublé 9 fois).

Idée de traduction (principe, pas la solution exacte) :

        MOV A, 1        ; A = x
        MOV B, 1        ; B = n
loop:   CMP B, 10       ; compare n et 10
        JAE end         ; si n >= 10, sortir
        INC B           ; n = n + 1
        ADD A, A        ; x = x + x
        JMP loop        ; recommencer
end:                    ; fin

À la fin de l'exécution, on retrouve A = 512 (valeur de x), comme attendu.

Dans le simulateur, on peut aussi stocker x et n en mémoire RAM via des labels (x: 0x0004, n: 0x0006, etc.), pour visualiser leur évolution dans la zone Memory.

1. Objectifs et définitions

1.1 Pourquoi le calcul parallèle ?

Le calcul séquentiel classique a atteint ses limites :

Limite

Constat

1. Temps d'exécution

Certains calculs prennent des heures, jours ou semaines : simulations météo, génomique, recherche médicale, IA…

2. Fréquence des processeurs

Augmenter la fréquence n'est plus une solution : mur de puissance (dissipation thermique trop importante)

3. Volume de données

Big data, images, vidéos explosent → le séquentiel devient un goulot d'étranglement

Conséquence : la fréquence d'horloge des processeurs stagne, voire diminue. Pour aller plus vite, il faut paralléliser.

1.2 Rappel : calcul séquentiel

Un cœur classique = Fetch/Decode + Execution Unit (ALU) + Execution Context (registres). Les instructions s'exécutent les unes après les autres.

ld   r0, addr[r1]
mul  r1, r0, r0
mul  r1, r1, r0
...
st   addr[r2], r0

Chaque instruction attend la fin de la précédente.

1.3 Définition du calcul parallèle

Un programme de calcul parallèle est un programme unique qui s'exécute simultanément sur plusieurs processeurs (ou cœurs).

C'est l'« Eldorado » de la conception : créer des machines puissantes en reliant un grand nombre de petits processeurs.

  • Les logiciels pour multiprocesseurs doivent être conçus pour fonctionner avec un nombre variable de processeurs

  • On parle de processeurs multicœurs : plusieurs cœurs dans une même puce

  • Le nombre de cœurs devrait continuer à augmenter

2. Les bases matérielles (rappels essentiels)

2.1 La mémoire : espace d'adressage

  • La mémoire est organisée comme un tableau de bytes (espace d'adressage)

  • Chaque byte est identifié par son adresse dans la mémoire

  • Exemple : un espace d'adressage de 32 bytes va de 0x00 à 0x1F

  • Le byte à l'adresse 0x8 peut contenir la valeur 32, celui à 0x10 la valeur 128, etc.

2.2 Les caches

  • Le cache est une mémoire intégrée au processeur (on-chip) qui conserve une copie d'un sous-ensemble des données présentes en mémoire principale

  • Si une adresse est présente dans le cache, le processeur lit/écrit beaucoup plus rapidement que si les données résidaient uniquement en DRAM

  • C'est un détail d'implémentation matérielle qui n'affecte pas le résultat d'un programme, uniquement ses performances (réduit la latence mémoire)

3. Les techniques de parallélisme (5 niveaux)

3.1 Le pipeline (rappel)

Une instruction « traverse » plusieurs étapes distinctes du chemin de données :

IF → ID → EX → MEM → WB

Avec un pipeline, plusieurs instructions peuvent être à des étages différents en même temps :

Cycle:    1    2    3    4    5    6    7    8    9
Instr 1:  IF   ID   EX   MEM  WB
Instr 2:       IF   ID   EX   MEM  WB
Instr 3:            IF   ID   EX   MEM  WB
Instr 4:                 IF   ID   EX   MEM  WB
Instr 5:                      IF   ID   EX   MEM  WB

Les étages doivent être séparés par des registres (ou latches) qui maintiennent les résultats intermédiaires.

3.2 Processeur superscalaire

Un processeur superscalaire est capable d'exécuter plusieurs instructions différentes en même temps, au cours d'un même cycle d'horloge.

C'est une caractéristique standard des processeurs modernes. Superscalaire ≠ multicœur : il s'agit d'exécuter plusieurs instructions sur un seul cœur.

Comment ?

  • Le cœur possède plusieurs unités de calcul (ALU) capables de travailler en parallèle

  • Si les instructions sont indépendantes, le processeur peut le détecter et en envoyer plusieurs simultanément vers ces unités

  • Une architecture superscalaire contient plusieurs pipelines en parallèle

Conditions de bon fonctionnement

  • Les instructions ne doivent pas utiliser la même ressource au même moment

  • L'une ne doit pas dépendre du résultat de l'autre

Limite du « tout multi-pipeline »

Passer à 4 pipelines complets dupliquerait trop de matériel. L'approche réelle : un seul pipeline équipé de plusieurs unités fonctionnelles (plusieurs ALU, FPU, unités vectorielles…). Les instructions sont réparties entre ces unités. Il faut donc maintenir un flux d'instructions élevé.

Exemple : le Core i7 est superscalaire 4-wide (jusqu'à 4 instructions par cycle si elles sont indépendantes).

3.3 Les processeurs multicœurs

Un cœur = un processeur complet (registres, CU, ALU, parfois caches) au sein d'une même puce. Une puce peut en contenir 2, 4, 8 ou plus.

Avantage

  • 2 cœurs peuvent calculer 2 éléments d'une fonction en parallèle

  • MAIS il faut l'indiquer aux processeurs multicœurs

Comment exprimer le parallélisme côté programmation ?

Via les threads logiciels :

Un thread logiciel est une unité d'exécution logicielle, créée par un programme, gérée par l'OS, et exécutée sur un cœur disponible.

Symétrique vs Asymétrique

  • Symétrique : tous les cœurs sont identiques

  • Asymétrique : certains cœurs sont dédiés (affichage, audio…)

3.4 Le parallélisme de données — SIMD

SIMD (Single Instruction, Multiple Data) : une seule instruction est diffusée vers toutes les ALU. Cette opération est exécutée en parallèle sur l'ensemble des ALU. La même instruction s'applique à plusieurs valeurs simultanément.

Idée n°2 : déplacer la complexité

Plutôt que gérer plusieurs flots d'instructions différents, on gère un seul flot mais sur un grand nombre d'ALU.

Comment l'utiliser ?

Type

Description

Implicite

Le programmeur n'écrit pas de code vectoriel. Le compilateur détecte des boucles simples et les transforme en instructions SIMD

Explicite

Le programmeur utilise un programme vectoriel (instructions SIMD) pour augmenter l'efficacité (ex. CUDA, MATLAB, OpenMP)

Limitation importante

⚠️ Toutes les données doivent subir la même opération au même moment !

→ SIMD est idéal pour les tableaux, matrices, images, signaux.

Registres vectoriels

Les registres vectoriels contiennent les opérandes destinés aux calculs vectoriels et leurs résultats. Leur taille détermine le niveau d'instructions SIMD supporté par le processeur.

AVX (Advanced Vector Extensions) = extension du jeu d'instructions x86 qui fournit des instructions SIMD au sein d'un même cœur.

3.5 Le multithreading

Le problème

Un ordinateur n'exécute jamais un seul programme à la fois. L'OS jongle en permanence avec des dizaines de threads logiciels issus de sources différentes.

Idée n°3

Entrelacer l'exécution de plusieurs threads logiciels sur un même cœur afin de masquer les stalls (blocages).

Si on ne peut pas progresser sur le thread courant (ex. attente d'une donnée en mémoire), on travaille sur un autre thread disponible.

C'est le multithreading matériel, aussi appelé SMT (Simultaneous Multi-Threading), ou Hyper-Threading chez Intel.

5. Architectures à grande échelle (HPC)

HPC (High-Performance Computing) : dans tout système parallèle, les CPU travaillant sur différentes parties du même programme doivent communiquer. Deux grandes conceptions selon le mode d'échange : mémoire partagée et mémoire distribuée.

5.1 Architecture à coprocesseurs

Un coprocesseur est un processeur supplémentaire ajouté pour des applications spécifiques :

  • GPU : calcul graphique et parallèle

  • DPU (Data Processing Unit) : processeurs réseau, gestion des paquets entrants/sortants en temps réel

5.2 Architecture à mémoire partagée

Principe

  • Plusieurs processeurs partagent un même espace mémoire

  • Toute modification dans la mémoire partagée est vue par tous les processeurs

  • Chaque processeur peut lire/écrire un mot mémoire via des instructions LOAD/STORE

  • Les threads communiquent en lisant et écrivant des variables dans l'espace d'adressage partagé

  • Tout processeur peut référencer directement n'importe quelle adresse mémoire

Exemple : serveur x86 Intel Xeon

  • Serveur SMP (Symmetric Multi-Processing) : ordinateur unique avec plusieurs processeurs partageant la mémoire principale, tous équivalents (même rôle, même accès mémoire)

  • Beaucoup de cœurs (threads)

  • Exemple : un serveur 4 processeurs × 16 cœurs = machine à 64 cœurs avec une seule mémoire

  • Interconnexion UPI (Ultra Path Interconnect) reliant processeurs et mémoire centrale

  • Très grande capacité mémoire (plusieurs To)

  • Les grands serveurs multiprocesseurs à mémoire partagée sont majoritairement bâtis sur Intel Xeon ou AMD EPYC, dans des configurations à 2, 4 ou 8 sockets

5.3 Architecture à mémoire distribuée

Principe

  • Chaque processeur (nœud) a sa propre mémoire privée

  • Chaque nœud accède à la mémoire des autres via un réseau dédié, en utilisant des messages

  • Pas de notion d'espace d'adressage global entre tous les CPU

  • Nécessite un mécanisme de synchronisation entre threads s'exécutant sur des machines différentes

  • Nœuds de calcul : exécutent les tâches en parallèle

  • Les centres de données pour l'IA, l'automatisation, l'analyse et le stockage explosent

  • NVIDIA et OpenAI : 100 milliards de dollars d'investissements pour des datacenters de 10 GWh (équivalent à dix centrales nucléaires)

  • 200 millions de milliards d'opérations par seconde

Aujourd'hui, il est possible de relier des organisations entières via Internet pour former des grilles de calcul.

6. À retenir

Synthèse des techniques

  • Le parallélisme se base sur la multiplication des cœurs de calcul

  • À l'intérieur du cœur : pipeline et architectures superscalaires (plusieurs unités fonctionnelles)

  • Au sein du cœur encore : SIMD/multithreading (un cœur gère plusieurs threads ou plusieurs données simultanément)

  • Sur la puce : multicœurs (plusieurs cœurs intégrés)

  • Au-delà : coprocesseurs (GPU, DPU…) ajoutent une capacité spécialisée

  • À grande échelle : multiprocesseurs à mémoire partagée (Intel Xeon, AMD EPYC) ou systèmes à mémoire distribuée (clusters, supercalculateurs, grilles)

Le goulot d'étranglement

Quelle que soit l'architecture, l'accès aux données et les transferts en mémoire constituent le goulot d'étranglement des architectures parallèles. C'est pour cela que les caches et les réseaux d'interconnexion sont aussi cruciaux que les cœurs eux-mêmes.

Continuer à améliorer les performances

Pour aller plus loin :

  • Utiliser l'exécution parallèle : multicœurs et SIMD

  • Réduire la latence mémoire : multithreading

  • Utiliser des accélérateurs : GPU

  • Utiliser le parallélisme intra-nœud : systèmes à mémoire partagée

  • Utiliser le parallélisme par échange de messages : systèmes à mémoire distribuée

Vue d'ensemble en une phrase

Pipeline + superscalaire + SIMD + multithreading → un cœur efficace. Multicœurs → une puce puissante. Multiprocesseurs + GPU → un serveur puissant. Clusters + supercalculateurs → de la puissance à l'échelle planétaire. -e

1. Introduction et objectifs

1.1 Pourquoi sauvegarder ?

Contexte actuel des entreprises :

  • Forte dépendance des entreprises envers leur infrastructure IT

  • Exigences de disponibilité et d'accessibilité aux données

  • Accroissement régulier du volume des données nécessaires

→ Exposition quotidienne à différents dangers (sinistre, panne, malveillance, erreur humaine…)

Principaux objectifs de la sauvegarde :

  • Continuité des activités

  • Protection contre la perte de données

  • Sauvegarde des données importantes

  • Compliance réglementaire

1.2 La sauvegarde, c'est quoi ?

Définition : la sauvegarde est une opération automatisée et planifiée visant à collecter des données, à mémoriser leur lieu de collecte, à les stocker de manière isolée, dans le but de les restaurer à l'identique en cas de perte ou de corruption.

Trois mots-clés à retenir : collecter, mémoriser, stocker (isolé).

2.1 Restauration

Opération déclenchée manuellement visant à restituer le contenu d'une ou des données ayant été précédemment sauvegardée(s) à un moment donné.

2.3 Types de données à sauvegarder

Données non structurées

  • Fichiers au format normalisé ou propriétaire

  • Textuels (bureautique) ou non textuels (vidéo, audio…)

  • ⚠️ Ne doivent pas être en cours d'utilisation lors de la sauvegarde (risque de corruption)

Données structurées

  • Bases de données et applications

  • Reposent ou non sur un système de fichiers

  • Format propriétaire

  • ⚠️ Si en cours d'utilisation, la sauvegarde doit se faire via des outils spécifiques (scripts, API…)

2.4 Cas particulier : export d'une base de données

  • Cas normal : export de la BD dans un fichier, puis sauvegarde du fichier sur le stockage secondaire

  • Cas où le fichier est trop volumineux : sauvegarde directe du flux d'export

2.5 Sauvegarde de disques (ou partitions)

La sauvegarde d'une partition disque ou d'un disque entier permet d'obtenir un clone de l'ensemble des données.

Objectif : restaurer rapidement à l'identique :

  • les données nécessaires à la production : répertoires utilisateurs, BD, comptes utilisateurs…

  • l'environnement technique de production : applications, partition système, table d'allocation…

Pour assurer l'intégrité des données, le système de sauvegarde doit comprendre un mécanisme de cliché instantané (snapshot) garantissant la mise à disposition des données à un instant donné.

2.6 Sauvegarde en mode Bloc

  • Technologie historiquement utilisée pour les bases de données (Oracle…)

  • Activation via des APIs (exemple : VMware vStorage API for Data Protection)

  • Principe : seuls les blocs modifiés sont collectés et transférés vers le stockage secondaire

  • Avantage majeur : forte diminution du volume de données sauvegardées

4. Paramètres techniques

4.1 Objectifs et principes

4.2 RPO — Recovery Point Objective

Unité de temps correspondant à la perte de données maximale admissible. Défini dans le passé à compter de l'instant où un incident s'est produit.

Unité de temps correspondant au délai maximal d'interruption admissible. Défini dans le futur à compter de l'instant où un incident s'est produit.

  • Plus le RTO est faible, plus vite l'entreprise peut reprendre ses activités après l'incident

4.5 Fenêtre de sauvegarde (Backup Window)

  • Créneau horaire durant lequel la sauvegarde des données peut se faire

  • Exprimé en unité de temps

  • Généralement organisé en dehors des heures de bureau

  • Systèmes critiques → fenêtre nulle (sauvegarde sans interruption de service)

4.6 Fréquence

Intervalle de temps entre deux sauvegardes.

  • Exprimée en unité de temps

  • Peut dépendre du type de sauvegarde

Exemple : 1 sauvegarde totale/semaine + 1 sauvegarde incrémentale/jour.

➡️ Période entre 2 sauvegardes ≤ RPO (sinon, on dépasse la perte de données acceptable)

4.7 Rétention

Durée de conservation continue des données sauvegardées (ou archivées) — souvent dictée par des contraintes légales de conformité.

  • Exprimée en unité de temps

  • Peut différer selon la nature des données

  • Peut différer selon le support de stockage secondaire

Paramétrage correct de la rétention

Rétention = fonction de (Fenêtre de restauration + Fréquence des Totales)

  • La fenêtre de restauration est la période pendant laquelle on peut restaurer une donnée

  • Les utilisateurs ne perçoivent pas toujours la rétention de la même façon que l'IT

4.8 SLA — Service Level Agreement

Un SLA est un contrat entre un prestataire de services et un client qui définit les niveaux de service attendus et les conditions de prestation.

Les RTO et RPO ont un impact direct sur le SLA. Plus ils sont bas, plus le SLA est exigeant (et coûteux).

6. À retenir

Les fondamentaux

  • Sauvegarder = collecter + mémoriser + stocker isolément des données pour pouvoir les restaurer à l'identique

  • ≠ Archiver : la sauvegarde sert à restaurer, l'archivage à retrouver

Les 3 types de sauvegardes

  • Complète : tout

  • Différentielle : depuis la dernière complète

  • Incrémentielle : depuis la précédente (peu importe son type)

Vue d'ensemble

La sauvegarde n'est pas un produit qu'on achète, mais une politique qu'on conçoit en équilibrant RPO, RTO, fenêtre, fréquence, rétention — selon les besoins métier, le coût, et les contraintes réglementaires. Veeam, Commvault, Veritas, Rubrik et autres outillent cette politique.
