Fichiers en C — Récapitulatif des exercices

Rappel des bases

Les 3 étapes obligatoires

Toute manipulation de fichier suit le même schéma :

  1. Ouvrir avec fopen

  2. Lire ou écrire avec fscanf, fgets ou fprintf

  3. Fermer avec fclose

Modes d'ouverture

Mode

Usage

"r"

Lecture — le fichier doit exister

"w"

Écriture — crée ou écrase le fichier

Vérification de fopen

fopen renvoie NULL en cas d'échec. Toujours vérifier avant de lire/écrire :

FILE* fichier = fopen("nom.txt", "r");
if (fichier == NULL) {
    printf("Impossible d'ouvrir le fichier.\n");
    return;
}

Exercice 1 — Transmission de données

Énoncé : lire un fichier caractère par caractère, afficher uniquement les chiffres et compter combien de données ont été correctement reçues. Le fichier contient un mélange de tirets et de chiffres comme --8---5-34--7--14.

Points clés à retenir

%c lit absolument tout — contrairement à %d ou %s qui sautent les espaces et retours à la ligne, %c prend le prochain caractère sans exception (tirets, espaces, \n...).

Le piège de feof — feof ne détecte la fin du fichier que après qu'une lecture a buté contre elle. Il faut donc :

  • Faire une lecture avant la boucle

  • Faire la lecture suivante à la fin de chaque tour

  • Sinon le dernier caractère est traité deux fois

Tester >= '0' et pas >= '1' — un zéro est une donnée valide, ne pas l'oublier.

Code commenté

#define _CRT_SECURE_NO_WARNINGS
#include <stdio.h>

void main(void) {

    FILE* fichier = fopen("transmission.txt", "r");

    if (fichier == NULL) {
        printf("Impossible d'ouvrir le fichier.\n");
        return;
    }

    int nbDonneesRecues = 0;
    char caractere;

    // Première lecture AVANT la boucle pour que feof
    // ait quelque chose à évaluer dès le premier tour
    fscanf(fichier, "%c", &caractere);

    while (!feof(fichier)) {

        // %c lit absolument TOUT (tirets, espaces, \n...)
        // contrairement à %d ou %s qui sautent les espaces

        // '0' à '9' couvre tous les chiffres valides
        if (caractere >= '0' && caractere <= '9') {
            printf("Donnee recue : %c\n", caractere);
            nbDonneesRecues++;
        }

        // Lecture suivante À LA FIN du tour pour que feof
        // soit évalué avec une lecture fraîche au prochain tour
        fscanf(fichier, "%c", &caractere);
    }

    fclose(fichier);

    printf("%d donnees recues\n", nbDonneesRecues);
}

Exercice 2 — Affichage avec numéros de ligne

Énoncé : programme en ligne de commande qui affiche le contenu d'un fichier texte avec ses numéros de ligne en vert. Style cat -n sous Linux.

Points clés à retenir

fgets au lieu de fscanf — quand les lignes contiennent des espaces, fscanf avec %s s'arrêterait au premier espace. fgets lit toute la ligne d'un coup.

fgets inclut le \n dans la chaîne lue, donc pas besoin d'ajouter \n dans le printf.

Codes couleur ANSI :

  • \033[1;32m → passe en vert

  • \033[0m → revient à la couleur normale

Ligne de commande — modifier la signature du main :

void main(int argc, char* args[])

  • argc = nombre de mots dans la commande (programme inclus)

  • args[0] = nom du programme

  • args[1], args[2]... = arguments fournis

C'est le système qui remplit argc et args[] automatiquement, on n'a rien à fournir.

Gestion d'erreurs en ordre :

  1. D'abord vérifier argc < 2 (sinon args[1] n'existe pas et fopen plante)

  2. Ensuite vérifier fichier == NULL

Deux versions pour la boucle de lecture

Version 1 (recommandée) — fgets comme condition

while (fgets(texte, 100, fichier) != NULL) {
    printf("\033[1;32m%d\033[0m  %s", numeroLigne, texte);
    numeroLigne++;
}

fgets renvoie NULL quand il n'y a plus rien à lire. Concis, robuste, gère le cas du fichier vide automatiquement.

Version 2 (alternative) — feof avec if après la boucle

fgets(texte, 100, fichier);

while (!feof(fichier)) {
    printf("\033[1;32m%d\033[0m  %s", numeroLigne, texte);
    numeroLigne++;
    fgets(texte, 100, fichier);
}

// La dernière ligne lue n'a pas été affichée dans la boucle,
// le if protège contre le cas où le fichier serait vide
if (texte[0] != '\0') {
    printf("\033[1;32m%d\033[0m  %s", numeroLigne, texte);
}

Plus verbeux, nécessite un if post-boucle pour gérer la dernière ligne. À utiliser si on veut garder le pattern feof cohérent avec d'autres parties du code.

Règle générale :

  • Tu utilises fgets → préfère sa valeur de retour comme condition

  • Tu utilises fscanf → feof est souvent plus naturel

Code commenté (étape finale avec gestion d'erreurs)

#define _CRT_SECURE_NO_WARNINGS
#include <stdio.h>

void main(int argc, char* args[]) {

    // Cas 1 : pas d'argument fourni
    // On vérifie AVANT fopen, sinon args[1] n'existe pas et le programme plante
    if (argc < 2) {
        printf("Erreur : aucun fichier fourni.\n");
        printf("Utilisation : affiche.exe nom_du_fichier\n");

    } else {

        FILE* fichier = fopen(args[1], "r");

        // Cas 2 : le fichier fourni n'existe pas
        if (fichier == NULL) {
            printf("Erreur : impossible d'ouvrir le fichier '%s'.\n", args[1]);

        } else {

            int numeroLigne = 1;
            char texte[100];

            // fgets renvoie NULL en fin de fichier — pas besoin de feof
            while (fgets(texte, 100, fichier) != NULL) {
                printf("\033[1;32m%d\033[0m  %s", numeroLigne, texte);
                numeroLigne++;
            }

            fclose(fichier);
        }
    }
}

Exercice 3 — Minigrep (analyse de logs)

Énoncé : créer un programme qui cherche un mot dans un fichier source et copie les lignes qui le contiennent dans un fichier résultat.

minigrep syslog Activating grepresult.txt

Points clés à retenir

Deux fichiers ouverts simultanément — un en lecture ("r"), un en écriture ("w"). Chaque FILE* a son propre curseur indépendant.

fprintf vers un fichier — exactement comme printf mais avec le fichier comme premier argument :

fprintf(destination, "%s", ligne);

Trois cas d'erreur à gérer dans l'ordre :

  1. argc < 4 — pas assez d'arguments

  2. source == NULL — fichier source inexistant

  3. destination == NULL — fichier résultat impossible à créer

Lire la signature de la fonction d'abord — quand on définit une fonction, sa signature (paramètres et type de retour) est le contrat. Si l'énoncé est ambigu sur ce que reçoit la fonction, choisir l'interprétation la plus simple. Si la fonction contientMot reçoit "une ligne", c'est le texte de la ligne, pas un numéro — sinon la fonction devrait gérer un fichier en plus, ce qui serait trop complexe pour son rôle.

Algorithme de contientMot

On parcourt la ligne caractère par caractère. À chaque position i, on teste si le mot correspond à partir de là, caractère par caractère avec un deuxième indice j.

ligne = "int printf(fichier, format);"
mot   = "printf"

i=0  → 'i' vs 'p' → non, on avance
i=1  → 'n' vs 'p' → non, on avance
...
i=4  → 'p' vs 'p' ✅ → on continue avec j
        'r' vs 'r' ✅
        'i' vs 'i' ✅
        ...
        mot[j] == '\0' → trouvé !

Code de la fonction (style propre, un seul return avec une vraie comparaison) :

bool contientMot(char ligne[], char mot[]) {

    int i = 0;
    int j = 0;

    // La boucle s'arrête si on est en fin de ligne OU si on a trouvé le mot
    while (ligne[i] != '\0' && mot[j] != '\0') {

        j = 0;
        // Comparaison caractère par caractère
        while (mot[j] != '\0' && ligne[i + j] == mot[j]) {
            j++;
        }

        // On n'avance dans la ligne que si on n'a pas trouvé le mot
        if (mot[j] != '\0') {
            i++;
        }
    }

    // Si j est arrivé au bout de mot → trouvé
    return mot[j] == '\0';
}

Code complet du minigrep

#define _CRT_SECURE_NO_WARNINGS
#include <stdio.h>
#include <stdbool.h>

bool contientMot(char ligne[], char mot[]) {
    int i = 0;
    int j = 0;
    while (ligne[i] != '\0' && mot[j] != '\0') {
        j = 0;
        while (mot[j] != '\0' && ligne[i + j] == mot[j]) {
            j++;
        }
        if (mot[j] != '\0') {
            i++;
        }
    }
    return mot[j] == '\0';
}

void main(int argc, char* args[]) {

    // Cas 1 : pas assez d'arguments
    if (argc < 4) {
        printf("Utilisation : <source> <mot a rechercher> <destination>\n");

    } else {

        FILE* source = fopen(args[1], "r");

        // Cas 2 : fichier source inexistant
        if (source == NULL) {
            printf("Impossible d'ouvrir %s\n", args[1]);

        } else {

            FILE* destination = fopen(args[3], "w");

            // Cas 3 : fichier résultat impossible à créer
            if (destination == NULL) {
                printf("Impossible de creer %s\n", args[3]);

            } else {

                char ligne[500];

                // Pour chaque ligne lue, on teste contientMot
                // et on copie dans le fichier résultat si trouvé
                while (fgets(ligne, 500, source) != NULL) {
                    if (contientMot(ligne, args[2])) {
                        fprintf(destination, "%s", ligne);
                    }
                }

                // On ferme les deux fichiers ouverts
                fclose(source);
                fclose(destination);
            }
        }
    }
}

Pièges à connaître

Le piège du feof

feof ne détecte la fin du fichier qu'après qu'une lecture a buté contre elle. Le pattern correct :

  • Une lecture avant la boucle

  • Une lecture à la fin de chaque tour

  • Sinon le dernier élément est traité deux fois

Le piège de %c vs %s vs %d

  • %c → lit absolument tout, espaces et \n compris

  • %s → s'arrête au premier espace

  • %d, %lf → sautent les espaces, cherchent le prochain nombre

Le piège de fgets et la dernière ligne

Avec le pattern feof, la dernière fgets lit la dernière ligne mais on n'entre plus dans la boucle pour l'afficher. Solutions :

  • Soit utiliser fgets() != NULL comme condition

  • Soit ajouter un affichage après la boucle avec un if de protection

Le piège de argc et args[1]

Si l'utilisateur ne fournit pas d'argument, args[1] n'existe pas. Toujours vérifier argc avant d'utiliser args[1] dans fopen, sinon le programme plante.

Le piège de l'interprétation de l'énoncé

Réflexe à développer :

  1. Lire la signature de la fonction d'abord (paramètres, retour)

  2. Reformuler l'énoncé avec ses propres mots

  3. Se demander : "existe-t-il une version plus simple de ce que j'ai compris ?"

  4. Si la solution semble très complexe, c'est souvent qu'on est sur la mauvaise piste

Rôle
