1 — Contexte et énoncé

Ce qui est demandé

  • Réseau de base : 172.16.0.0/16

  • VLSM avec les nombres d'hôtes présents sur la topologie, sans oublier les réseaux inter-routeurs

  • Les PC et serveurs seuls prennent la première adresse disponible, les gateways des routeurs prennent la dernière

  • R1 et S1 doivent être configurés dans les règles de l'art

  • Les VLANs doivent communiquer entre eux

  • Le routage entre tous les réseaux se fait via RIP v2

  • Le serveur Web est accessible via une route par défaut propagée par RIP v2

Hypothèse de travail (zone VLAN 10 / VLAN 20)

L'étiquette "2000 hôtes" sur la zone VLAN 10 / VLAN 20 a été interprétée comme 1000 hôtes par VLAN (total partagé). Si l'énoncé d'examen est différent, recalculer le VLSM en conséquence.

Vérification d'alignement

Une adresse réseau valide doit avoir tous ses bits hôte à 0. Concrètement :

  • Un /22 doit démarrer sur un multiple de 4 dans le 3ᵉ octet (0, 4, 8, 12...)

  • Un /26 sur un multiple de 64 dans le 4ᵉ octet (0, 64, 128, 192)

  • Un /27 sur un multiple de 32 (0, 32, 64, 96, 128, 160, 192, 224)

  • Un /30 sur un multiple de 4 (0, 4, 8, 12...)

Le tri décroissant garantit l'alignement automatique : chaque gros bloc tombe naturellement sur un point de départ valide pour le suivant.

Logique générale

6 — Phase 2 — Configuration des routeurs

Concept 1 — DCE / DTE et clock rate

Sur un lien Serial, l'un des deux routeurs (le DCE) doit fournir l'horloge qui synchronise la communication. Dans la vraie vie c'est l'opérateur télécoms ; en simulation c'est le côté du câble qui porte l'icône ⌚.

clock rate 64000

Cette commande est obligatoire uniquement côté DCE. Le côté DTE n'a rien à faire.

Concept 2 — Router-on-a-stick et encapsulation dot1q

Quand un routeur sert de gateway pour plusieurs VLANs via une seule interface physique, on crée des sous-interfaces numérotées (g0/0/0.10, g0/0/0.20, etc.). Chaque sous-interface est associée à un VLAN via encapsulation dot1q <vlan>.

L'interface physique doit être allumée (no shutdown) avant de créer les sous-interfaces.

Côté switch, le port qui mène au routeur doit être en trunk pour conserver les tags 802.1q.

configure terminal

interface gig0/0/0

interface gig0/0/0.30

 ip address 172.16.11.254 255.255.252.0
 exit

7 — Phase 3 — RIP v2

Concepts

Sans protocole de routage, chaque routeur ne connaît que ses réseaux directement connectés. RIP permet aux routeurs d'échanger automatiquement leurs tables de routage.

Trois commandes clés à connaître :

  • version 2 : utilise RIP v2 (supporte le VLSM, contrairement à v1)

  • no auto-summary : désactive le résumé automatique en classes (indispensable avec VLSM)

  • network <adresse-classful> : déclare les réseaux directement connectés à annoncer

Règle importante : on ne déclare que les réseaux directement connectés, jamais les réseaux distants — RIP les apprend tout seul.

Router2

Deux réseaux directement connectés : le VLAN 30 bas et les liens Serial.

enable
configure terminal

router rip
 version 2
 no auto-summary
 network 172.16.11.0
 network 172.16.12.0
 exit

end

8 — Phase 4 — Route par défaut

Le concept

Le serveur WEB est dans un réseau qui n'est pas annoncé via RIP. Pour que les autres routeurs puissent l'atteindre, R1 doit :

  1. Créer une route statique par défaut vers le serveur WEB

  2. Propager cette route par défaut dans RIP via default-information originate

Tous les routeurs apprennent alors une route 0.0.0.0/0 qui pointe vers R1 — ils savent que "tout ce qu'ils ne connaissent pas" doit transiter par lui.

Configuration (sur R1 uniquement)

enable
configure terminal

ip route 0.0.0.0 0.0.0.0 172.16.12.97

router rip
 default-information originate
 exit

end
copy run start

Vérification

Sur un routeur distant (Router5 par exemple) :

show ip route

Tu dois voir :

  • Une ligne commençant par R* (la route par défaut apprise via RIP)

  • En haut : Gateway of last resort is <ip> to network 0.0.0.0

10 — Phase 6 — Tests de connectivité

La cascade

À effectuer depuis un PC (ex: PC2 dans le VLAN 10). On s'arrête au premier test qui rate.

1. Ping intra-VLAN     : PC2 → autre PC du VLAN 10
2. Ping inter-VLAN     : PC2 → PC du VLAN 20
3. Ping inter-routeur  : PC2 → PC du VLAN 30 bas
4. Ping serveur WEB 2  : PC2 → 172.16.12.65 (valide RIP)
5. Ping serveur WEB    : PC2 → 172.16.12.97 (valide la default route)

À noter : le premier paquet d'un ping est souvent perdu (timeout) le temps que ARP résolve l'adresse MAC. C'est normal et ne signale pas un problème.

Diagnostic si un test échoue

show ip interface brief   ! Toutes les interfaces utilisées en up/up ?
show ip route             ! Le routeur connaît-il le réseau destination ?
show ip protocols         ! RIP est-il bien activé ?
show vlan brief           ! (sur switch) le port est bien dans le bon VLAN ?
show interfaces trunk     ! (sur switch) les trunks sont bien up ?

Pourquoi le SVI et la default-gateway sur S1

Un switch est un équipement de couche 2 — il n'a pas besoin d'IP pour faire son boulot de commutation. Mais pour pouvoir le pinguer, y accéder en SSH/Telnet, ou le gérer à distance, il lui faut une adresse IP.

  • SVI (Switched Virtual Interface) : interface vlan 30 crée une interface virtuelle dans le VLAN 30 qui porte une IP.

  • ip default-gateway : indique au switch où envoyer ses réponses quand elles doivent sortir de son VLAN. Sans cette commande, S1 ne peut être pingué que depuis le VLAN 30.

13 — Concepts à maîtriser

13.1 — Conversion binaire / décimal

Une adresse IPv4 fait 32 bits = 4 octets de 8 bits chacun. Chaque octet va de 0 à 255.

Les 8 positions d'un octet valent toujours :

128 — 64 — 32 — 16 — 8 — 4 — 2 — 1

Chaque position est le double de la suivante (en partant de la droite).

Méthode de conversion décimal → binaire : pour chaque position de gauche à droite, demande "est-ce que mon nombre est ≥ à cette valeur ?". Si oui, bit = 1 et on soustrait. Sinon, bit = 0.

Exemple pour 192 :

  • 192 ≥ 128 ? Oui → bit = 1, reste 64

  • 64 ≥ 64 ? Oui → bit = 1, reste 0

  • Tout le reste = 0

  • Résultat : 11000000 = 192 ✓

13.3 — Bits réseau vs bits hôte

  • Bits réseau : fixés par le masque (les 1s). Dans une adresse réseau, ils définissent l'identité du réseau.

  • Bits hôte : libres (les 0s du masque). Ils identifient chaque machine dans le réseau.

  • Adresse réseau : tous les bits hôte à 0 (réservée, non assignable à une machine).

  • Adresse broadcast : tous les bits hôte à 1 (réservée).

  • IPs utilisables = 2ⁿ − 2 où n est le nombre de bits hôte.

13.4 — Alignement des sous-réseaux

Chaque sous-réseau doit démarrer sur un point aligné en fonction de son masque. Concrètement, l'octet variable doit être un multiple du pas :

  • /22 → 3ᵉ octet multiple de 4

  • /26 → 4ᵉ octet multiple de 64

  • /27 → 4ᵉ octet multiple de 32

  • /30 → 4ᵉ octet multiple de 4

Le tri décroissant en VLSM garantit l'alignement automatique des gros blocs.

13.5 — DCE / DTE sur lien Serial

Sur un lien Serial entre routeurs, il faut un signal d'horloge pour synchroniser la communication. Dans la vraie vie c'est l'opérateur télécoms qui le fournit ; en simulation c'est l'un des deux routeurs (le DCE, identifié par l'icône ⌚) qui s'en charge via clock rate 64000.

13.6 — Router-on-a-stick

Un routeur peut servir de gateway à plusieurs VLANs via une seule interface physique en utilisant des sous-interfaces avec encapsulation dot1q <vlan>. Le port du switch en face doit être en trunk pour conserver les tags 802.1q.

13.7 — RIP v2

Protocole de routage à vecteur de distance qui annonce les réseaux directement connectés à ses voisins. Avec VLSM, toujours utiliser version 2 + no auto-summary.

La commande network <adresse> annonce le réseau et active RIP sur les interfaces correspondantes — uniquement pour les réseaux directement connectés.

13.8 — Route par défaut propagée

Quand un réseau doit être atteignable mais n'est pas dans RIP (cas du serveur WEB ici), on crée une route statique par défaut sur le routeur qui y a accès, puis on la propage via default-information originate dans RIP.

14 — Pièges courants

Configuration de base

  1. Oublier no shutdown sur une interface — elle reste DOWN.

  2. Oublier copy run start — tout perdu au reload.

  3. Oublier delete vlan.dat au reset d'un switch — les VLANs ressuscitent.

Adressage

  1. Oublier le −2 dans le calcul des hôtes (réseau + broadcast).

  2. Mettre une IP dans la plage réservée (.0 ou .255 d'un /24).

  3. Mauvais masque parce qu'on a oublié qu'un /28 fait 14 IPs (pas 16).

VLANs

  1. Oublier encapsulation dot1q X sur les sous-interfaces.

  2. Numéro dot1q qui ne correspond pas au numéro de VLAN sur le switch.

  3. Oublier switchport mode access sur un port — il reste en dynamic auto.

  4. Oublier switchport mode trunk sur le port qui mène au routeur.

  5. Oublier d'allumer l'interface physique avant les sous-interfaces.

Routage

  1. RIP : oublier no auto-summary avec VLSM — les routes sont résumées en /16.

  2. RIP : annoncer un réseau distant dans network au lieu d'un réseau directement connecté.

  3. Tester avant la convergence (environ 30 secondes pour RIP).

Default route

  1. Oublier default-information originate — la route par défaut ne se propage pas.

  2. Annoncer le réseau du serveur WEB dans RIP — la route par défaut n'a plus de sens.

Sur lien Serial

  1. Oublier clock rate côté DCE — le lien reste down/down.

  2. Mettre clock rate des deux côtés — ne fonctionne pas non plus.

Sur S1

  1. Oublier no shutdown sur le SVI — il reste DOWN même avec une IP.

  2. Oublier ip default-gateway — le switch n'est joignable que depuis son VLAN.
