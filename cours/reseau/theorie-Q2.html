1 — Introduction aux réseaux

1.1 — Pourquoi des réseaux ?

Un réseau permet à plusieurs équipements de partager des ressources (fichiers, imprimantes, connexion Internet) et de communiquer. Aujourd'hui, les réseaux sont au cœur de presque toutes nos activités : banque en ligne, météo, streaming, communication, travail collaboratif.

1.2 — Types de réseaux

LAN — Local Area Network

Un réseau local dessert une zone géographiquement limitée : une maison, un bâtiment ou un campus. C'est un réseau privé, géré par son propriétaire, avec des débits élevés (10 Mbps à plusieurs Gbps). L'équipement représentatif est le switch.

WAN — Wide Area Network

Un réseau étendu connecte des LANs séparés géographiquement (ex : sites d'une entreprise dans plusieurs villes ou pays). Il est généralement public (opéré par un FAI), avec des débits plus lents que les LANs. L'équipement représentatif est le routeur.

1.3 — Caractéristiques d'une bonne architecture réseau

Quatre qualités fondamentales définissent un réseau fiable :

Tolérance aux pannes (Fault tolerance) — des connexions redondantes permettent des chemins alternatifs si un lien ou équipement tombe. L'utilisateur ne perçoit pas la panne.

Évolutivité (Scalability) — l'architecture doit pouvoir grandir sans nécessiter une refonte complète.

Qualité de service (QoS) — priorisation du trafic critique. Par exemple, dans un réseau convergé (voix + vidéo + données), la VoIP a priorité sur les transactions financières, qui ont elles-mêmes priorité sur le trafic web.

Sécurité — repose sur trois piliers :

  • Confidentialité : authentification des utilisateurs + chiffrement des données

  • Intégrité : signatures numériques pour garantir que les données n'ont pas été altérées

  • Disponibilité : firewalls, redondance, architecture sans point unique de défaillance

2 — Modèles en couches : OSI et TCP/IP

2.1 — Pourquoi un modèle en couches ?

Un modèle en couches est une représentation conceptuelle qui décompose la complexité de la communication réseau en niveaux indépendants. Chaque couche a une responsabilité précise et communique avec les couches adjacentes via des interfaces standardisées.

Avantages :

  • Aide à la conception des protocoles

  • Favorise la concurrence entre constructeurs (interopérabilité)

  • Les changements dans une couche n'affectent pas les autres

  • Fournit un langage commun pour discuter du réseau

Le modèle est seulement une représentation, pas le réseau lui-même.

2.4 — Encapsulation

Lorsqu'un message est envoyé, chaque couche ajoute son en-tête aux données reçues de la couche supérieure. À la réception, le processus inverse (décapsulation) se produit.

Exemple : pour envoyer une requête HTTP,

Données HTTP                                  ← Couche 7
[En-tête TCP][Données HTTP]                   ← Couche 4 (Segment)
[En-tête IP][En-tête TCP][Données HTTP]       ← Couche 3 (Paquet)
[En-tête Eth][En-tête IP][...][CRC]           ← Couche 2 (Trame)
01010111010001011...                          ← Couche 1 (Bits)

3 — Couche 1 — Physique

La couche physique transmet les bits bruts sur le support : cuivre, fibre optique ou ondes radio. Elle définit les caractéristiques électriques, mécaniques et de timing du média.

3.1 — La fibre optique

La fibre optique transmet l'information sous forme d'impulsions lumineuses dans un fil de verre ou de plastique. Avantages :

  • Très haut débit (jusqu'à plusieurs Tbps)

  • Distance importante sans répéteur (km)

  • Insensible aux interférences électromagnétiques

  • Sécurité accrue (difficile à intercepter)

Elle est utilisée à la fois en backbone WAN, en accès Internet (FTTH) et de plus en plus dans les LANs d'entreprise.

4 — Couche 2 — Liaison de données

4.1 — Rôle de la couche 2

La couche liaison de données prépare les données pour la transmission sur un média physique spécifique. Son travail principal est de gérer l'accès au média, ce qui implique :

  1. S'adapter au support — chaque tronçon peut utiliser une technologie différente (Ethernet, Wi-Fi, fibre…). À chaque saut, l'équipement intermédiaire décapsule la trame et la réencapsule dans le format approprié au support suivant.

  2. Mettre en œuvre une stratégie d'accès au média :

    • Maître-esclave : une machine gère le droit de parole

    • Passage du jeton : chaque machine parle à son tour

    • Accès libre (CSMA/CD) : on parle quand on veut, on retransmet en cas de collision

  3. Fournir un connecteur physique (carte réseau)

4.3 — Ethernet

Ethernet est la technologie LAN dominante. Elle s'étend des couches 1 (802.3) et 2 (LLC 802.2 + MAC).

Évolution topologique

Historiquement Ethernet utilisait un bus physique partagé (tous les ordinateurs sur un même câble). Aujourd'hui :

  • Topologie physique en étoile (chaque PC connecté à un switch central)

  • Topologie logique en bus sur les anciens hubs (concentrateurs) → le hub diffuse à tous

  • Topologie complète en étoile avec les switchs modernes → trafic dirigé uniquement vers le destinataire

4.4 — Adresses MAC

Une adresse MAC (Media Access Control) est un identifiant unique sur 48 bits (6 octets en hexadécimal), gravé dans la carte réseau à la fabrication.

 00:60:2F:3A:07:BC
└──OUI───┘└──NIC──┘
24 bits     24 bits

  • OUI (Organizationally Unique Identifier) : identifie le constructeur (ex : 00:60:2F = Cisco)

  • NIC : identifiant unique de l'équipement attribué par le constructeur

4.5 — Le switch et la table de commutation

Le switch est l'équipement central des LANs Ethernet modernes. Il opère à la couche 2 et utilise les adresses MAC pour aiguiller les trames.

Construction dynamique de la table MAC

Au démarrage, la table MAC est vide. Le switch apprend en observant le trafic :

  1. Learning — quand une trame arrive sur un port, le switch lit l'adresse MAC source et l'enregistre dans sa table avec le numéro de port.

  2. Flooding — si l'adresse MAC destination est inconnue, le switch envoie la trame sur tous les ports sauf le port d'entrée.

  3. Selective Forwarding — si l'adresse MAC destination est connue, la trame est envoyée uniquement sur le bon port.

Exemple concret :

État initial : MAC TABLE vide

Étape 1 — Host1 (MAC 0A) sur Fa1 envoie à Host2 (MAC 0C)
  → Switch enregistre : Fa1 = 0A
  → Switch ne connaît pas 0C : flooding sur Fa3, Fa6, Fa8
  → Seul Host2 accepte la trame (les autres l'ignorent)

Étape 2 — Host2 (0C) sur Fa6 répond à Host1 (0A)
  → Switch enregistre : Fa6 = 0C
  → Switch connaît 0A (Fa1) : forwarding sélectif sur Fa1

4.6 — Types de communication Ethernet

  • Unicast : un à un (MAC destination spécifique)

  • Multicast : un à plusieurs (groupe identifié)

  • Broadcast : un à tous (MAC destination = FF:FF:FF:FF:FF:FF)

Un broadcast de couche 2 est limité au domaine de broadcast. Les routeurs ne propagent pas les broadcasts de couche 2.

4.7 — ARP — Address Resolution Protocol

ARP fait le lien entre la couche 3 (IP) et la couche 2 (MAC). Pour envoyer un paquet IP à une machine du même LAN, il faut connaître son adresse MAC.

Fonctionnement :

  1. La machine A veut joindre l'IP 192.168.1.10. Elle ne connaît pas sa MAC.

  2. A envoie un ARP Request en broadcast : "qui a l'IP 192.168.1.10 ?"

  3. La machine ayant cette IP répond par un ARP Reply en unicast : "c'est moi, voici ma MAC"

  4. A garde la correspondance dans son cache ARP pour les communications suivantes.

4.8 — VLAN — Virtual LAN

Un VLAN segmente logiquement un switch en plusieurs réseaux indépendants, alors que physiquement il n'y en a qu'un.

Avantages

  • Réduit la taille des domaines de broadcast — moins de trafic inutile

  • Classe le trafic par fonction, département ou type de données

  • Améliore la sécurité — les VLANs sont isolés les uns des autres

  • Virtualise l'infrastructure en segments logiques

Implémentation des VLANs

  • Statique : un port = un VLAN, configuration manuelle (méthode classique)

  • Dynamique : attribution selon la MAC ou l'IP du périphérique (plus coûteuse, utilisée en sécurité avancée)

Modes des ports

Access port :

  • Appartient à un seul VLAN

  • Connexion typique : un PC, un téléphone IP

  • Aucun TAG ajouté aux trames

  • Transporte le trafic de plusieurs VLANs

VTP — Virtual Trunking Protocol

Protocole Cisco qui synchronise les VLANs entre tous les switches d'un domaine.

  • 3 rôles : serveur (un seul), client, transparent (ne fait que transmettre)

  • Nom de domaine VTP commun à tous les switches

  • VTP Pruning : évite l'envoi de broadcasts vers des switches qui n'ont pas le VLAN concerné

  • Les messages VTP fonctionnent par numéro de révision (le plus élevé est pris en compte)

⚠️ VTP est déconseillé en production : un switch fraîchement configuré avec un numéro de révision élevé peut écraser toute la configuration VLAN du domaine.

STP — Spanning Tree Protocol

Dans un réseau avec des liens redondants (pour la tolérance aux pannes), il existe des boucles physiques qui peuvent provoquer des tempêtes de broadcast. STP détecte ces boucles et désactive logiquement certains liens, en gardant un arbre couvrant sans boucle. Si un lien actif tombe, STP réactive un lien désactivé pour maintenir la connectivité.

4.9 — Router-on-a-stick

Configuration permettant à un seul routeur de router le trafic inter-VLAN via une seule interface physique configurée en trunk. On crée des sous-interfaces (ex : Gi0/0.10, Gi0/0.20), une par VLAN, chacune avec sa propre IP servant de gateway pour son VLAN.

5 — Couche 3 — Réseau (IPv4)

5.1 — Rôle de la couche réseau

La couche réseau permet à des périphériques situés sur des réseaux différents de communiquer. Elle s'occupe de :

  • L'adressage logique (IP) indépendant du média

  • Le routage entre réseaux via les routeurs

  • L'encapsulation des PDU de transport en paquets IP

Le PDU de cette couche dans les réseaux TCP/IP est le paquet IP : [En-tête IP][PDU de la couche transport].

5.2 — Caractéristiques d'IP

1. Sans connexion

IP n'établit aucune session avant d'envoyer un paquet. C'est l'analogie d'une lettre postale : on la dépose dans la boîte sans savoir si le destinataire est présent. L'émetteur ne sait pas si le récepteur est là, si la lettre est arrivée, ou s'il peut la lire.

2. Best effort (au mieux, non fiable)

IP ne garantit rien : pas de retransmission, pas d'accusé de réception, pas de réordonnancement. Les paquets peuvent être perdus, dupliqués ou arriver dans le désordre. La fiabilité est gérée par la couche supérieure (TCP) si nécessaire.

IP n'est pas un mauvais protocole — c'est un choix de design qui le rend rapide et léger. La fiabilité est ajoutée quand on en a besoin par TCP.

3. Indépendant du média

IP fonctionne sur n'importe quelle technologie de couche 2 : Ethernet, Wi-Fi, fibre, satellite, etc.

5.3 — Adressage IPv4

Une adresse IPv4 est codée sur 32 bits, écrite en notation décimale pointée : 192.168.1.10. Elle se compose de deux parties :

  • Net-Id : identifie le réseau

  • Host-Id : identifie la machine dans ce réseau

La séparation entre les deux est définie par le masque de sous-réseau.

Le masque de sous-réseau

Le masque sépare la partie réseau de la partie hôte. Il est composé d'une suite continue de 1 (partie réseau) suivie de 0 (partie hôte).

Exemple : IP 195.145.123.32 avec masque 255.255.255.0 (notation CIDR /24)

  • Les 24 premiers bits = réseau → 195.145.123.0

  • Les 8 derniers bits = hôte → machine n°32

L'opération est un AND logique entre l'IP et le masque pour extraire l'adresse réseau.

6 — Subnetting et VLSM

6.1 — Pourquoi créer des sous-réseaux ?

Découper un réseau en sous-réseaux apporte plusieurs bénéfices :

  • Réduction des domaines de broadcast — moins de trafic inutile

  • Sécurité — on peut isoler des groupes de machines

  • Organisation — alignement avec la structure de l'entreprise (services, étages…)

  • Économie d'adresses — utilisation plus efficace de l'espace d'adressage disponible

6.2 — Subnetting classique

On emprunte des bits au Host-Id pour créer des sous-réseaux. Les bits empruntés deviennent le Subnet-Id.

Formules :

  • Nombre de sous-réseaux = 2ⁿ (n = bits empruntés)

  • Nombre d'hôtes utilisables par sous-réseau = 2ᵐ − 2 (m = bits restants pour les hôtes ; le -2 correspond à l'adresse du réseau et de broadcast)

Exemple pratique

192.168.30.170 avec masque 255.255.255.224 (/27)

IP     :  11000000.10101000.00011110.10101010
Masque :  11111111.11111111.11111111.11100000
                                    ───┘└──┘
                                 Subnet  Host
AND    :  11000000.10101000.00011110.10100000

  • Réseau de classe C : 192.168.30.0

  • Sous-réseau : 101 en binaire = sous-réseau n°5

  • Machine : 01010 = machine n°10 dans ce sous-réseau

Méthode pour calculer un masque

À partir d'un nombre de sous-réseaux voulu :

  1. Trouver le plus petit n tel que 2ⁿ ≥ nombre de sous-réseaux

  2. Emprunter n bits au Host-Id par défaut

À partir d'un nombre d'hôtes max par sous-réseau :

  1. Trouver le plus petit m tel que 2ᵐ - 2 ≥ nombre d'hôtes

  2. Garder m bits pour les hôtes, le reste forme le masque

6.3 — VLSM — Variable Length Subnet Mask

Le subnetting classique impose un masque unique → tous les sous-réseaux ont la même taille. Mais souvent, les besoins sont très différents (un siège avec 500 PCs vs un bureau avec 20 personnes).

VLSM permet d'utiliser des masques variables : chaque sous-réseau a la taille adaptée à son besoin, ce qui économise les adresses.

Règle d'or : allouer les plus grands blocs en premier

Exemple : réseau 172.16.0.0/22 à partager entre :

  • Corporate HQ : 500 hôtes → /23 (510 hôtes utiles) → 172.16.0.0/23

  • Sales Office : 200 hôtes → /24 (254 hôtes utiles) → 172.16.2.0/24

  • HR Office : 50 hôtes → /26 (62 hôtes utiles) → 172.16.3.0/26

  • Legal Office : 20 hôtes → /27 (30 hôtes utiles) → 172.16.3.64/27

  • Liens WAN (2 hôtes chacun) → /30 (2 hôtes utiles) → blocs suivants

Sans VLSM, on aurait dû utiliser /23 pour tout le monde et gaspiller des centaines d'adresses sur les petits sites.

7 — Routage

7.1 — Rôle du routeur

Le routeur est l'équipement de couche 3 qui relie des réseaux différents. Quand un paquet arrive avec une IP destination qui n'est pas dans le réseau local, il est envoyé à la passerelle (gateway) — c'est-à-dire au routeur.

7.2 — Une route — 3 composants

Une route est une entrée dans la table de routage. Elle est composée de :

  1. Réseau de destination (IP de réseau)

  2. Masque (longueur du préfixe)

  3. Gateway / Next-hop : l'IP du prochain routeur à qui transmettre le paquet

Sur un routeur Cisco, on consulte la table de routage avec show ip route.

Exemple de table de routage Cisco

R 10.1.1.0/24 [120/1] via 192.168.2.2, 00:00:08, FastEthernet0/0
R 10.1.2.0/24 [120/1] via 192.168.2.2, 00:00:08, FastEthernet0/0
C 192.168.2.0/24 is directly connected, FastEthernet0/0

  • R = route apprise via RIP (dynamique)

  • C = réseau directement connecté

  • [120/1] = distance administrative / métrique

  • via 192.168.2.2 = next-hop

7.3 — Principes fondamentaux du routage

  1. Chaque routeur décide seul, basé uniquement sur sa propre table de routage.

  2. Un routeur ne connaît pas toute la topologie — il sait juste où envoyer le paquet pour le rapprocher de la destination.

  3. L'information de retour n'est pas garantie — le chemin retour peut être différent.

  4. La table de routage applique le principe de longest match : la route ayant la plus grande correspondance (préfixe le plus long) est choisie.

7.4 — Routage statique

L'administrateur configure manuellement chaque route sur chaque routeur.

Avantages :

  • Facile à configurer pour un petit réseau

  • Pas de ressources supplémentaires (pas de protocole qui tourne)

  • Plus sécurisé (pas de propagation automatique)

Inconvénients :

  • Pas adapté aux grandes topologies

  • Tout changement nécessite une reconfiguration manuelle

  • Évolutivité limitée

  • Risque de boucles de routage si mal configuré

Recursive Route Lookup

Quand un routeur cherche une route, il peut faire plusieurs tours dans sa table : si la première route trouvée pointe vers un next-hop qui n'est pas directement connecté, il cherche à nouveau pour trouver comment joindre ce next-hop.

Bonne pratique : spécifier le next-hop plutôt que l'interface (sauf pour les liens série point-à-point).

Liens série : DTE et DCE

  • DTE (Data Terminal Equipment) : reçoit le clock — pas de clockrate à configurer

  • DCE (Data Communication Equipment) : fournit le clock — clockrate obligatoire

7.5 — Route par défaut

ip route 0.0.0.0 0.0.0.0 [next-hop]

Cette route capture tout le trafic dont la destination ne correspond à aucune autre route. Le masque 0.0.0.0 correspond à n'importe quelle adresse (en AND, le résultat est toujours 0.0.0.0).

C'est la route de dernier recours, typiquement utilisée pour pointer vers le FAI.

7.6 — Résumé de route (route summarization)

Combiner plusieurs routes statiques en une seule route plus large.

3 étapes :

  1. Supprimer les routes statiques existantes

  2. Configurer les routes résumées

  3. Vérifier le bon fonctionnement

Avantages : table de routage plus petite → plus rapide à parcourir, moins de mémoire utilisée.

Inconvénient : ne supporte pas la discontinuité (les sous-réseaux résumés doivent être contigus).

7.7 — Routage dynamique

Les routeurs s'échangent automatiquement des informations sur les réseaux qu'ils connaissent via un protocole de routage.

Avantages :

  • S'adapte aux changements (lien qui tombe, nouveau réseau ajouté)

  • Scalable

  • Optimisation automatique des chemins

Inconvénients :

  • Consomme plus de ressources (CPU, RAM, bande passante)

  • Plus complexe à configurer

  • Moins sécurisé (les annonces peuvent être interceptées ou usurpées)

8 — NAT — Network Address Translation

8.1 — Pourquoi le NAT ?

L'espace d'adressage IPv4 (4,3 milliards d'adresses) est insuffisant pour le nombre de machines connectées. Le NAT permet à de nombreuses machines avec des IP privées de partager une (ou quelques) IP publique(s) pour accéder à Internet.

8.2 — Types de NAT

NAT statique

Association 1-1 entre une IP privée et une IP publique. Toujours la même IP publique pour la même machine privée.

  • Usage : exposer un serveur interne avec une IP publique fixe

  • Inconvénient : nécessite autant d'IPs publiques que de machines

NAT dynamique

Un pool d'adresses publiques est partagé entre les machines internes. À chaque connexion sortante, une IP publique du pool est attribuée.

  • Usage : entreprises avec plusieurs IPs publiques

  • Inconvénient : si toutes les IPs du pool sont utilisées, les nouvelles connexions échouent

NAT Overload (PAT — Port Address Translation)

Le plus utilisé. Plusieurs (souvent toutes les) IPs privées partagent une seule IP publique, différenciées par les numéros de port.

Avantages :

  • Conservation maximale des IPs publiques

  • Facile à mettre en place

  • Transparent pour les utilisateurs internes

Exemple PAT

Machine interne A : 192.168.5.2:1056 → traduit en 28.2.4.6:1056
Machine interne B : 192.168.5.3:1055 → traduit en 28.2.4.6:1055

Le routeur garde une table de correspondance :
  192.168.5.2:1056 ↔ 28.2.4.6:1056
  192.168.5.3:1055 ↔ 28.2.4.6:1055

Quand une réponse arrive sur 28.2.4.6:1056, le routeur sait qu'elle est destinée à 192.168.5.2:1056.

8.3 — Avantages et inconvénients du NAT

Avantages :

  • Économie massive d'IPs publiques

  • Configuration simple côté client

  • Sécurité implicite (les machines internes ne sont pas joignables directement depuis Internet)

Inconvénients :

  • Problèmes avec certains protocoles (FTP en mode actif, IPsec, SIP)

  • Complexité du débogage

  • Casse le principe end-to-end d'IP

9 — ICMP — diagnostic et signalisation

9.1 — Rôle d'ICMP

ICMP (Internet Control Message Protocol) est un protocole de signalisation de couche 3, encapsulé dans IP.

Caractéristiques :

  • Caractère optionnel — non nécessaire à la transmission

  • Complète le protocole IP (qui n'a aucun mécanisme de signalisation)

  • Permet de signaler les erreurs de transmission

  • Permet de diagnostiquer des problèmes réseau

  • Certains messages sont automatiques (ex : routeur renvoyant une erreur)

  • D'autres sont déclenchés par des commandes (ex : ping)

9.4 — Commandes basées sur ICMP

ping (types 8 et 0)

Vérifie la connectivité entre deux équipements. Le PING teste les 3 premières couches OSI (physique, liaison, réseau).

Émetteur                              Destinataire
  ──── ICMP Echo Request (type 8) ──→
  ←──── ICMP Echo Reply (type 0) ────

tracert / traceroute (type 11)

Détermine le chemin emprunté par un paquet en exploitant le TTL.

Mécanisme :

  1. La source envoie un paquet avec TTL = 1

  2. Le premier routeur décrémente le TTL à 0, supprime le paquet et renvoie un ICMP Time Exceeded → la source apprend l'IP du 1er routeur

  3. La source ré-envoie avec TTL = 2 → 2ème routeur identifié

  4. Et ainsi de suite jusqu'à atteindre la destination

9.5 — ICMP Redirect (type 5)

Un routeur envoie un Redirect à un hôte pour lui indiquer qu'il existe une meilleure route vers une destination.

Exemple :

  • L'hôte H a R1 comme passerelle par défaut

  • H envoie un paquet à R1 pour atteindre 10.1.1.1

  • R1 voit que le next-hop est R2, sur la même interface que H

  • R1 transfère le paquet à R2 et envoie un ICMP Redirect à H

  • H utilise désormais R2 directement

Conditions pour un Redirect :

  1. L'interface d'entrée = l'interface de sortie

  2. Source et next-hop sont sur le même sous-réseau

  3. Le paquet n'est pas source-routed

  4. ICMP Redirects activés (par défaut sur Cisco)

9.6 — ICMP et dépannage

3 méthodes selon le modèle OSI pour diagnostiquer un problème :

Par le bas (méthode ascendante) — partir de la couche physique. Bonne approche si on suspecte un problème matériel (câble débranché, port défectueux). Inconvénient : long, chaque équipement doit être vérifié.

Par le haut (méthode descendante) — partir de l'application. Bonne approche pour des problèmes liés à une application précise. Inconvénient : peut être long si plusieurs applications sont impactées.

Par le milieu — récolter des informations auprès de l'utilisateur, documenter les symptômes, choisir la couche de départ selon le diagnostic. Si la couche choisie fonctionne, on peut généralement supposer que les couches inférieures vont bien et remonter. Sinon, on descend une à une.

10 — IPv6

10.1 — Pourquoi IPv6 ?

IPv4 offre théoriquement 4,3 milliards d'adresses. Avec l'explosion d'Internet (objets connectés, mobiles, etc.), cet espace est épuisé. Tous les RIRs (Regional Internet Registries) ont commencé à manquer d'adresses entre 2015 et 2020.

IPv6 apporte :

  • Adresses sur 128 bits → 340 undecillion (3,4 × 10³⁸) adresses

  • Suppression des limitations d'IPv4

  • ICMPv6 enrichi (NDP)

  • Configuration automatique simplifiée (SLAAC)

10.2 — Coexistence IPv4 / IPv6

3 techniques de migration :

Dual-stack (meilleure solution) — les équipements font tourner IPv4 et IPv6 simultanément. Chaque paquet utilise le protocole adapté.

Tunneling — encapsule un paquet IPv6 dans un paquet IPv4 pour traverser un réseau IPv4 uniquement. Utile pour interconnecter deux îlots IPv6 via Internet (encore majoritairement IPv4).

Translation (NAT64) — traduit les paquets IPv6 ↔ IPv4 (similaire au NAT classique). Permet à des équipements IPv6 de communiquer avec des équipements IPv4.

10.3 — Notation d'une adresse IPv6

128 bits écrits en hexadécimal, en 8 groupes (hextets) de 16 bits séparés par : :

2001:0DB8:0000:0000:0000:FF00:0042:8329

Règles de simplification

Règle 1 — Omettre les zéros de tête dans chaque hextet :

01AB → 1AB
0A00 → A00
00AB → AB

Règle 2 — Double deux-points :: : remplace un seul groupe de hextets entièrement à zéro consécutifs (utilisable une seule fois par adresse).

2001:0DB8:0000:0000:0000:FF00:0042:8329
↓
2001:DB8::FF00:42:8329

Préfixe

Pas de masque en notation décimale pointée. Le préfixe indique la partie réseau :

2001:DB8:ACAD::/48

Longueur typique du préfixe utilisateur : /64.

  • Subnet ID : 0001 → un sous-réseau interne du site

  • Interface ID : ::0010 → identifie l'interface de la machine

10.6 — Attribution d'une adresse IPv6

Configuration statique

L'admin configure manuellement l'adresse, le préfixe et la passerelle.

SLAAC — Stateless Address Autoconfiguration

Méthode sans serveur DHCPv6 qui repose sur les messages ICMPv6 Router Advertisement (RA).

Fonctionnement :

  1. La machine envoie un Router Solicitation (RS) en multicast à tous les routeurs (FF02::2)

  2. Les routeurs répondent avec un Router Advertisement (RA) contenant le préfixe, la longueur de préfixe et la gateway

  3. La machine génère son Interface ID (EUI-64 ou aléatoire) et compose son adresse IPv6

EUI-64 — Génération automatique de l'Interface ID

Construit l'Interface ID (64 bits) à partir de la MAC (48 bits) :

  1. Inverser le 7ème bit de l'OUI (Universal/Local bit)

  2. Insérer FFFE au milieu des 48 bits MAC

Exemple :

MAC : 00:60:2F:3A:07:BC

Étape 1 — inverser bit 7 du 1er octet
  00 = 00000000 → 00000010 = 02

Étape 2 — insérer FFFE au milieu
  02:60:2F:FF:FE:3A:07:BC

Interface ID : 0260:2FFF:FE3A:07BC

Avantage : la MAC est traçable dans l'IP. Inconvénient : problème de vie privée → Windows Vista+ génère un Interface ID aléatoire par défaut.

DHCPv6

Similaire à DHCP en IPv4. Deux modes :

  • Stateless DHCPv6 : combiné avec SLAAC. SLAAC fournit l'adresse, DHCPv6 fournit les infos complémentaires (DNS, NTP…)

  • Stateful DHCPv6 : DHCPv6 fournit tout (adresse + autres paramètres)

Les 3 options annoncées par le RA

  • Option 1 — SLAAC Only : "j'ai tout ce qu'il te faut (préfixe, longueur, gateway)"

  • Option 2 — SLAAC + DHCPv6 : "j'ai l'adressage, demande le DNS au serveur DHCPv6"

  • Option 3 — DHCPv6 Only : "je ne peux rien faire, demande tout au DHCPv6"

10.7 — Adresse Link-Local

Toute interface IPv6 a automatiquement une adresse link-local (FE80::/10) après l'assignation d'une adresse globale. Elle :

  • Permet la communication sur le même lien uniquement

  • N'est pas routée

  • Est utilisée comme next-hop dans les tables de routage IPv6

  • Sert pour les protocoles de routage dynamique IPv6

10.8 — ICMPv6 et NDP (Neighbor Discovery Protocol)

ICMPv6 est plus puissant qu'ICMPv4 : il remplace ARP via le protocole NDP, qui définit 4 nouveaux messages :

Message

Rôle

Router Solicitation (RS)

L'hôte cherche un routeur (multicast vers FF02::2)

Router Advertisement (RA)

Le routeur annonce ses paramètres (préfixe, gateway)

Neighbor Solicitation (NS)

Résolution d'adresse (remplace ARP) ou DAD

Neighbor Advertisement (NA)

Réponse à une NS

DAD — Duplicate Address Detection

Avant d'utiliser une adresse, la machine envoie un NS avec sa propre adresse comme cible. Si quelqu'un répond, l'adresse est déjà utilisée → conflit détecté.

11 — Couche 4 — Transport (TCP/UDP)

11.1 — Rôle de la couche transport

La couche transport prépare les données applicatives pour le réseau et gère la communication de processus à processus (entre applications).

11.2 — Fonctionnalités communes à TCP et UDP

Segmentation — découper les données en morceaux transportables sur le réseau.

Multiplexage — plusieurs applications peuvent utiliser le réseau simultanément. C'est le port qui distingue les flux.

Contrôle d'erreurs — checksum pour vérifier l'intégrité des données reçues.

11.3 — Les ports

Le port identifie l'application destinataire sur la machine. Une connexion est définie par 4 éléments : IP source : port source ↔ IP destination : port destination.

11.4 — TCP — Transmission Control Protocol

TCP est orienté connexion et fiable. Défini dans la RFC 793.

Caractéristiques principales :

  • Fiabilité via accusés de réception (ACK) et retransmission des segments perdus

  • Ordre garanti par numérotation et réassemblage

  • Connexion établie via le 3-way handshake

  • Contrôle de flux avec fenêtre glissante

  • Stateful : garde une trace de l'état de la session

Flags TCP

Flag

Signification

SYN

Synchronize — ouverture de connexion

ACK

Acknowledge — l'acknowledgement number est valide

FIN

Finish — fermeture propre de la connexion

RST

Reset — fermeture brutale (erreur)

PSH

Push — vider le buffer côté récepteur

URG

Urgent — données prioritaires

Numéros de séquence et d'acquittement

  • Sequence Number (SEQ) — numéro du premier octet de données du segment

  • Acknowledgement Number (ACK) — numéro du prochain octet attendu par le récepteur

Quand un segment est envoyé, il est copié dans une file d'attente avec un timer. Si pas d'ACK avant la fin du timer → retransmission.

TCP est full-duplex : chaque côté est à la fois émetteur et récepteur.

Établissement de la session — Three-Way Handshake

A → B : SYN=1, ACK=0, SeqNum=300
B → A : SYN=1, ACK=1, SeqNum=100, AckNum=301
A → B : SYN=0, ACK=1, SeqNum=301, AckNum=101

Décryptage :

  1. A demande l'ouverture, propose son numéro de séquence initial (300)

  2. B accepte (SYN+ACK), propose son numéro de séquence initial (100), confirme avoir reçu jusqu'à 301 (= 300+1)

  3. A confirme avoir reçu jusqu'à 101 (= 100+1) → la connexion est établie

Fin de la session

Fermeture en 4 étapes (chaque côté ferme indépendamment) :

A → B : FIN
B → A : ACK
B → A : FIN
A → B : ACK

Reset de la session

Si un problème survient, RST ferme brutalement la connexion :

A → B : ACK=1, RST=0, SeqNum=200, AckNum=400
B → A : ACK=0, RST=1, SeqNum=400

Contrôle de flux — fenêtre glissante

Si l'émetteur devait attendre un ACK après chaque segment, la transmission serait très lente. La fenêtre glissante permet d'envoyer plusieurs segments avant d'attendre un ACK.

La taille de la fenêtre est négociée et peut s'ajuster dynamiquement (réduction en cas de pertes, augmentation en cas de succès).

11.5 — UDP — User Datagram Protocol

UDP est sans connexion et non fiable (best effort).

Caractéristiques :

  • Pas de handshake, pas d'établissement de session

  • Pas d'accusés de réception

  • Pas de retransmission

  • Pas de contrôle de flux

  • Pas d'ordre garanti

  • En-tête minimal (8 octets) → faible surcharge

Idéal pour : VoIP, streaming vidéo, DNS, DHCP, TFTP, jeux en ligne — applications où la vitesse prime sur la fiabilité.

12 — Couches 5-7 — Application

La couche application (qui regroupe les couches OSI 5, 6 et 7) est l'interface entre les utilisateurs/applications et le réseau. C'est là que vivent les protocoles que l'utilisateur final connaît : HTTP pour le web, SMTP pour le mail, DNS pour la résolution de noms…

Ce chapitre couvre les principaux protocoles applicatifs :

  • DNS — résolution de noms

  • HTTP — web

  • DHCP — attribution dynamique d'IP

  • Mail — SMTP, POP3, IMAP

  • SMB — partage de fichiers

  • FTP / TFTP — transfert de fichiers

12.1 — DNS — Domain Name Service

DNS traduit les noms de domaine (ex : www.google.com) en adresses IP et inversement.

Ordre de résolution sur un client

  1. Cache local — résolutions récentes mémorisées

  2. Fichier HOST (Linux : /etc/hosts, Windows : C:\Windows\System32\drivers\etc\hosts)

  3. Serveurs DNS configurés sur la machine

Structure du DNS — base de données distribuée et hiérarchique

Le DNS est organisé en arbre :

                .   ← Racine
                |
   ─────────────────────────
   |     |     |     |     |
  com   org   edu    fr    be   ← TLD
   |
  google                        ← Domaine
   |
  www                           ← Hôte

Le nom complet d'une machine en partant de la feuille jusqu'à la racine est appelé FQDN (Fully Qualified Domain Name) : www.google.com.

Concepts clés :

  • Chaque nœud est un domaine ; chaque feuille est un hôte

  • Chaque nœud a une étiquette (label)

  • Chaque domaine est géré par un serveur DNS (serveur de noms)

  • Chaque entité est responsable de la gestion de son nom de domaine

Zones DNS

Une zone est un découpage administratif du namespace. Chaque zone a un fichier de zone (la base de données de cette zone). Un serveur DNS peut gérer plusieurs zones.

Délégation de zone : une zone peut déléguer la gestion d'une sous-zone à un autre serveur. Exemple : la gestion de iesn.be est déléguée par les serveurs de .be au serveur DNS de l'IESN.

Types de requêtes DNS

DNS utilise le port 53 :

  • UDP pour les requêtes classiques

  • TCP pour les transferts de zones (synchronisation master/slave)

Requête itérative (typique entre serveurs) : "donne-moi la meilleure réponse que tu as, même partielle". Le client se débrouille pour suivre les indications.

Requête récursive (typique d'un client vers son DNS) : "trouve-moi la réponse complète, ou dis-moi qu'elle n'existe pas". Le serveur DNS doit aller chercher l'info pour le client.

Requêtes inverses : IP → nom (utilisent les enregistrements PTR).

Types de réponses

  • Autoritative — provient d'un serveur qui gère officiellement la zone (master ou slave)

  • Non autoritative — provient d'un serveur qui n'est pas autoritatif mais a la réponse en cache

Exemple de résolution complète

Client veut résoudre www.bonjourponey.fr.iesn.be

1.  Client          → DNS local        : "www.bonjourponey.fr.iesn.be ?"
2.  DNS local       → Redirecteur      : "résous-moi ça"
3.  Redirecteur     → Serveur racine   : "qui gère .be ?"
4.  Serveur racine  → Redirecteur      : "voici les NS de .be"
5.  Redirecteur     → Serveur .be      : "qui gère iesn.be ?"
6.  Serveur .be     → Redirecteur      : "voici les NS de iesn.be"
7.  Redirecteur     → Serveur iesn.be  : "donne-moi www.bonjourponey.fr.iesn.be"
8.  Serveur iesn.be → Redirecteur      : "voici l'IP"
9.  Redirecteur     → DNS local        : "voici l'IP"
10. DNS local       → Client           : "voici l'IP"

12.2 — HTTP — HyperText Transfer Protocol

HTTP est le protocole du web. Il transporte du texte (HTML) et des fichiers (via les types MIME).

HTTP ≠ HTML, et Internet ≠ Web. Le Web a été co-inventé par un belge (Robert Cailliau, avec Tim Berners-Lee).

Caractéristiques

  • Modèle client-serveur

  • Port 80 (HTTP), port 443 (HTTPS), port 8080 (proxy classique)

  • Non connecté — chaque requête est indépendante (sauf option keep-alive)

  • Versions : HTTP/1.1 (RFC 2616), HTTP/2, HTTP/3 (2019)

12.3 — DHCP — Dynamic Host Configuration Protocol

DHCP automatise l'attribution d'adresses IP et de paramètres réseau (masque, gateway, DNS) aux machines.

Modes d'allocation

Allocation statique — association d'une MAC ou d'un nom d'hôte avec une IP fixe. Toujours la même IP pour la même machine.

Allocation dynamique — IP attribuée à partir d'un pool, pour une durée limitée (le bail). Avantages : centralisé, partage optimal, pas de conflits, idéal pour les clients mobiles.

Cycle de vie du bail DHCP

Phase

Description

Affectation

Acquisition initiale du bail

Réallocation

Au redémarrage avec un bail encore valide, le client demande à conserver ses anciens paramètres

Opération normale

Utilisation pendant T1 (50% du bail)

Renouvellement

À T1, tente de renouveler auprès du même serveur (jusqu'à T2 = 87,5%)

Réaffectation

À T2, tente auprès de n'importe quel autre serveur DHCP

Libération

Le client renonce volontairement au bail

Exemple avec un bail de 8 jours :

  • T1 = 4 jours → début des tentatives de renouvellement

  • T2 = 7 jours → début des tentatives de réaffectation

  • T = 8 jours → expiration

Les 4 étapes d'initialisation (DORA)

  1. DHCPDISCOVER — le client diffuse en broadcast pour trouver les serveurs DHCP disponibles

  2. DHCPOFFER — chaque serveur disponible répond avec une offre (IP candidate + paramètres)

  3. DHCPREQUEST — le client choisit une offre et le confirme en broadcast (les autres serveurs voient leur offre déclinée)

  4. DHCPACK — le serveur sélectionné confirme et envoie la configuration finale

Client                                     Serveur DHCP
  │                                              │
  │ ──── DHCPDISCOVER (broadcast) ─────────────→│
  │                                              │
  │ ←──── DHCPOFFER (unicast) ──────────────────│
  │                                              │
  │ ──── DHCPREQUEST (broadcast) ──────────────→│
  │      "j'ai étudié votre offre, je l'accepte" │
  │                                              │
  │ ←──── DHCPACK (unicast) ─────────────────────│
  │      "c'est bon, voici la config"            │

Autres messages DHCP

  • DHCPNAK — le serveur refuse (ex : IP demandée déjà allouée, bail expiré, client sur mauvais subnet)

  • DHCPDECLINE — le client refuse (ex : il a détecté que l'IP est déjà utilisée via ARP)

  • DHCPRELEASE — le client libère volontairement le bail

Format d'un message DHCP

L'en-tête comporte des champs comme :

  • Code OP : type d'opération

  • CIADDR : Client IP Address

  • YIADDR : Your IP Address (l'IP attribuée au client)

  • SIADDR : Server IP Address

  • GIADDR : Gateway IP Address (relais BOOTP/DHCP)

  • CHADDR : Client Hardware Address (MAC)

  • Options DHCP (variable)

Ports DHCP

  • Serveur : port UDP 67

  • Client : port UDP 68

Cas particulier du DHCPREQUEST

Le DHCPREQUEST est utilisé dans 3 cas (affectation, renouvellement, réaffectation). Ce qui change :

  • Les champs remplis par le client (notamment CIADDR)

  • Les adresses source/destination

  • Le serveur interprète selon le contexte

  1. Connexion au serveur (port 110)

  • Utilisateur sur un seul poste

  • Garantit que la boîte serveur ne déborde pas

  • Disponibilité hors connexion

Des copies locales restent possibles pour la consultation hors connexion.

12.5 — SMB — Server Message Block

SMB est un protocole client/serveur pour partager des ressources : fichiers, imprimantes, ports série.

Caractéristiques :

  • Développé par IBM à la fin des années 1980

  • Connexion persistante : après authentification, l'utilisateur peut accéder aux ressources distantes comme si elles étaient locales

  • Format de message commun : en-tête fixe + paramètres et données variables

Implémentations :

  • Microsoft : avant Windows 2000, basé sur NetBIOS pour la résolution de noms ; à partir de Windows 2000, sur DNS

  • Linux/Unix : Samba (réimplémentation libre, interopérable avec Windows)

12.6 — FTP — File Transfer Protocol

FTP transfère des fichiers entre client et serveur.

Caractéristiques :

  • Transfert dans les deux sens (upload / download)

  • Modification du contenu d'un répertoire (créer / supprimer)

  • Très fiable, avec reprise après erreur

  • Intéressant pour les gros fichiers

  • Très performant (transferts parallèles, compression)

Inconvénient : peut être compliqué à utiliser derrière un firewall à cause du fonctionnement à 2 ports.

Principe de fonctionnement

FTP utilise deux ports :

  • Port 21 — contrôle : commandes et réponses

  • Port 20 — données : transferts de fichiers

Versions sécurisées

  • FTPS : FTP sur SSL/TLS

  • SFTP : FTP sur SSH (totalement différent dans son fonctionnement, malgré le nom)

Clients FTP

  • Ligne de commande (ftp en console)

  • Navigateurs web (ftp://...)

  • Clients graphiques : FileZilla, WinSCP
